Key relationships:

- *sage* is both the MCP host and MCP client in a single process.
- *Ollama* is orthogonal to MCP. Ollama returns =tool_calls= arrays;
  sage routes each tool to either its local tool registry or to an MCP
  server via =mcp-call-tool=.
- The MCP server *pushes* JSON-RPC responses over SSE; the HTTP POST
  returns only =202 Accepted=. Request/response correlation is by
  JSON-RPC =id= across two separate TCP connections.
- Telemetry is fire-and-forget. The REPL never blocks or crashes on
  telemetry failure.

** Tool-call extraction detail

Ollama returns =tool_calls= in the =message= object. The field may be
a JSON array (list from the JSON parser) or a vector (if sage built
it). =ollama-parse-tool-call= uses =as-list= from =util.scm= to
normalize both shapes, then extracts =function.name= and
=function.arguments=. Fallback: if =tool_calls= is absent, parses
=```tool\n{...}```= blocks from content text.

** Permission check flow

#+BEGIN_SRC
execute-tool(name, args)
  │
  ├─ name ∈ *safe-tools*?  → ALLOW
  ├─ SAGE_YOLO_MODE set?   → ALLOW
  └─ else                  → DENY (log WARN, inc reject counter)

** Path containment (=safe-path?=)

- Absolute paths starting with =/tmp/= are always allowed.
- Relative paths are resolved against =(workspace)= via =resolve-path=.
- Blocked patterns: =..=, =.env=, =.git/=, =.ssh=, =.gnupg=.
- =canonicalize-path-safe= handles symlinks.

** MCP tool policy

MCP tools are registered via =register-tool= with =safe: #f= (the
default). They are never added to =*safe-tools*=. Therefore *all MCP
tool invocations require YOLO mode* --- or the model gets a "Permission
denied" response and can reformulate.

This is a deliberate security posture: MCP tools execute arbitrary
operations on remote servers. The user must opt in.

* 7. MCP Client Architecture

** Transport: curl + FIFO workaround

The MCP SSE transport cannot use Guile's native =(web client)= due to
two platform bugs:

1. *Accept header rejection*: Guile's HTTP header validator rejects
   =Accept: text/event-stream= as an invalid MIME type.

2. *open-input-pipe broken on macOS*: =(open-input-pipe "curl ...")=
   fails with "Bad file descriptor in execvp" on macOS.

The workaround creates a FIFO (named pipe) at =/tmp/sage-sse-<pid>=,
runs =curl -sN= in the background writing to the FIFO, and opens the
FIFO as an input file port. This synchronizes reader and writer
without polling.

#+BEGIN_SRC
    sse-open-connection(url, headers)
      │
      ├── mkfifo /tmp/sage-sse-<pid>
      ├── curl -sN -H 'Accept: text/event-stream' <url> > /tmp/sage-sse-<pid> &
      └── (open-input-file "/tmp/sage-sse-<pid>")  → blocking until curl writes
          │
          └── returns: input port for SSE stream
#+END_SRC

** Lifecycle

1. =mcp-init= reads =~/.claude.json= to discover SSE MCP servers.
2. For each server: open SSE connection, wait for =endpoint= event,
   send =initialize= JSON-RPC, wait for SSE response, send
   =notifications/initialized=.
3. =mcp-discover-tools!= sends =tools/list=, parses the response.
4. =mcp-register-tools-from-list= registers each tool under both
   a namespaced name (=<server>.<tool>=) and a bare name (if no
   collision with built-in tools).
5. Tool invocation via =mcp-call-tool=: JSON-RPC =tools/call= POST,
   wait for SSE response, extract =result.content[].text=.

** Correlation model

HTTP POST (JSON-RPC request) and SSE GET (JSON-RPC response) are
separate TCP connections. Correlation is by JSON-RPC =id=
(monotonically increasing integer from =*mcp-next-id*=).

** Invariant reference

See =docs/MCP-CONTRACT.org= for the full set of numbered invariants
(I01--I23) covering transport, lifecycle, tool discovery, and error
handling.

through =util.scm= is logged as a JSONL line to =.logs/http.jsonl=:

- Request: timestamp, method, URL, headers (Authorization redacted),
  body (truncated to 8192 chars unless =SAGE_DEBUG_HTTP_FULL=1=).
- Response: timestamp, URL, status code, elapsed_ms, body.
- Streaming: chunk_count, first_chunk, last_chunk.

This enables bidirectional protocol verification without rebuilding
requests manually.

* 9. Model-Tier Auto-Selection

#+BEGIN_SRC
resolve-model-for-tokens(current_tokens, available_tiers)
  │
  ├── tokens < fast_ceiling (2000)?     → fast tier (llama3.2:latest)
  ├── tokens < standard_ceiling (6000)? → standard tier (qwen2.5-coder:7b)
  └── else                              → last tier (heaviest available)
#+END_SRC

Tiers are configurable via env vars:

| Env var                        | Default            |
|--------------------------------+--------------------|
| =MODEL_TIER_FAST=              | =llama3.2:latest=  |
| =MODEL_TIER_STANDARD=          | =qwen2.5-coder:7b= |
| =MODEL_TIER_CEILING_FAST=      | 2000               |
| =MODEL_TIER_CEILING_STANDARD=  | 6000               |

At startup, =load-model-tiers= cross-references configured tiers
against =ollama-list-models= output and filters to only available
models. Each tier carries a =tools?= flag; the REPL only switches to
tiers that support native tool calling.

Context limits per model are resolved via =*token-limits*= in
=config.scm= (pattern-match on model name substring).

* 10. Design Philosophy: Tools Over Memory

Sage is NOT tool-agnostic. It explicitly uses file-based tools rather
than in-memory state.

| In-Memory (avoided)           | File-Based Tools (preferred)      |
|-------------------------------+-----------------------------------|
| TODO list in session state    | =bd create/list/close=            |
| Notes stored in variables     | =git notes=, =write_file=         |
| Context in conversation only  | =AGENTS.md= loaded at startup     |
| Lost on restart               | Persisted in filesystem            |

** The AGENTS.md contract

On startup, sage reads =AGENTS.md= (searched in CWD, project XDG,
global XDG, =~/.sage/=) and injects the combined contents as a system
message. This tells the model how to use the task system, project
workflows, and beads conventions.

* 13. Configuration Precedence

=config-get= resolves values in this order:

1. =SAGE_<KEY>= env var (uppercased)
2. =<KEY>= env var (uppercased)
3. Hash table entry from =config-load-dotenv=
4. Default value (caller-provided)

Dotenv files loaded from: =.env= (project), =~/.config/sage/.env=
(user).

XDG directories follow the freedesktop.org spec with sage-specific
subdirectories under each base:

#+BEGIN_SRC
~/.config/sage/     (config)
~/.local/share/sage/ (data: sessions, agents, projects, commands)
~/.cache/sage/      (cache)
~/.local/state/sage/ (state)
#+END_SRC
