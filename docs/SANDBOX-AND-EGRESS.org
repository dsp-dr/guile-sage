#+TITLE: guile-sage: Sandboxing, egress, and connection modes
#+OPTIONS: toc:2

Sage talks to one model backend at a time, selected by =SAGE_PROVIDER=. You can
run it directly on the host, or confined in a sandbox. This note covers the three
connection modes, how to run sage in a sandbox, and the egress each mode needs.

* Connection modes

Configured via =SAGE_PROVIDER= and the matching vars (see =.env.template= for the
full set). Three shapes, cheapest-isolation first:

| Mode | =SAGE_PROVIDER= | Endpoint var | Secret in env | Egress target |
|------+-----------------+--------------+---------------+---------------|
| Direct Ollama (local) | =ollama= | =SAGE_OLLAMA_HOST= (default =http://localhost:11434=) | none | localhost only |
| Via LiteLLM gateway | =openai= | =SAGE_OPENAI_BASE= (e.g. =http://<litellm-host>:4000/v1=) | *proxy key only* | the proxy host:port |
| Direct provider | =gemini= / =anthropic= / =openai= | provider default / =SAGE_OPENAI_BASE= | *real provider key* | provider API host |

**Prefer the LiteLLM gateway** when you want one endpoint, cost metering,
guardrails, and — critically — to keep *provider* secrets out of the sage process.
In gateway mode sage holds only the low-privilege proxy key (=SAGE_OPENAI_API_KEY=
= the proxy key), never the upstream provider key; the proxy injects the real key
at egress. This is the property that makes sandboxing meaningful.

#+begin_src sh
# Direct Ollama (local, no key)
SAGE_PROVIDER=ollama SAGE_OLLAMA_HOST=http://localhost:11434

# Via LiteLLM (proxy key only; model name selects the upstream)
SAGE_PROVIDER=openai \
SAGE_OPENAI_BASE=http://<litellm-host>:4000/v1 \
SAGE_OPENAI_API_KEY=sk-litellm-local \
SAGE_MODEL=gemini-2.5-flash        # or ollama-llama3.2, gemini-2.5-pro, ...

# Direct provider (real key in env — only when not going through the gateway)
SAGE_PROVIDER=gemini GEMINI_API_KEY=...
#+end_src

* Running in a sandbox

The sage process is a good sandbox candidate: give it the model endpoint and the
*one* key that mode needs, deny everything else. Pick the boundary per platform.

** macOS — =sandbox-exec= (seatbelt), headless, no daemon

Deny reads of credential stores and writes into =$HOME=; run with a *keyless*
environment (=env -i= → only the proxy key reaches sage). Profile:

#+begin_src scheme
;; sage-sandbox.sb
(version 1)
(allow default)
(deny file-read*
  (subpath "/Users/<you>/.aws") (subpath "/Users/<you>/.config/gcloud")
  (subpath "/Users/<you>/.ssh") (subpath "/Users/<you>/.ollama")
  (regex #"/\.env$"))
(deny file-write* (subpath "/Users/<you>"))
(allow file-write* (subpath "/tmp") (subpath "/private/var/folders"))
#+end_src

#+begin_src sh
sandbox-exec -f sage-sandbox.sb env -i \
  HOME=/Users/<you> PATH=/opt/homebrew/bin:/usr/bin:/bin GUILE_AUTO_COMPILE=0 \
  SAGE_PROVIDER=openai SAGE_OPENAI_BASE=http://<litellm-host>:4000/v1 \
  SAGE_OPENAI_API_KEY=sk-litellm-local SAGE_MODEL=gemini-2.5-flash \
  SAGE_DATA_DIR=/tmp/sage SAGE_CACHE_DIR=/tmp/sage/cache SAGE_CONFIG_DIR=/tmp/sage/config \
  guile -L src -l scripts/<your-entry>.scm
#+end_src

Verified: confined sage lists gateway models but is *denied* the credential store
(=.env= read → =system-error=); unconfined it can read it. Seatbelt host-level
network filtering is coarse, so the keyless =env -i= is what guarantees the
credential property — pair them.

** FreeBSD — Bastille jail (validated)

Run sage in a Bastille jail; restrict egress with =pf= NAT so the jail can reach
only the gateway host:port. The jail has no access to host credential files — it
holds only the proxy key. Native, no container runtime.

Validated full stack: *Bastille jail → pf NAT → LiteLLM proxy (<litellm-host>:4000)
→ Gemini*, with guile-sage generating a haiku about jails from inside a jail. The
credential boundary held: the jail never saw a provider secret.

The pattern is language-agnostic — anything that can HTTP POST to the proxy works
from inside the jail. Runtimes validated against the gateway from a jail:

| Runtime      | Tool          | Status                          |
|--------------+---------------+---------------------------------|
| Guile Scheme | guile-sage    | ✓ haiku generated               |
| Emacs Lisp   | efrit         | ✓ working                       |
| Clojure      | nREPL (nexus) | ✓ (shared heap, not jailed yet) |

This is the core point: *the proxy is the universal credential boundary*; the jail
(or seatbelt/container) is just the execution boundary around whatever language
makes the call.

** Linux / container — Docker, podman, or =colima= (headless macOS)

Run sage in a container whose network policy permits only the gateway. Mount no
host secrets; pass the proxy key via env.

** Process-level — Deno / WASM (for tool code)

For untrusted *tool* code rather than sage itself: Deno
(=--allow-net=<litellm-host>:4000=, no =--allow-env=) or Pyodide (WASM, host-mediated
fetch) give a keyless, network-pinned execution context.

* Egress configuration

Allow the minimum per mode. Two layers when sandboxed: the *sandbox/host* network
policy, and (if applicable) the LiteLLM credential's own allowed-hosts.

| Mode | Allow egress to |
|------+-----------------|
| Direct Ollama | =localhost:11434= (no external egress) |
| LiteLLM gateway | the proxy only, e.g. =<litellm-host>:4000= |
| Direct Gemini | =generativelanguage.googleapis.com:443= |
| Direct Anthropic | =api.anthropic.com:443= |
| Direct OpenAI | =api.openai.com:443= |

Gateway mode collapses egress to a single internal host — the simplest allowlist,
and the upstream provider hosts are reached by the *proxy*, not by sage.
