* 1. Goal & threat model

Run guile-sage so that a *compromised or prompt-injected agent cannot exfiltrate
anything durable*. Two invariants:

1. *No long-lived credential ever enters the sandbox.* The agent holds only an
   *ephemeral, scoped LiteLLM virtual key* — short TTL, capped budget, limited
   to its team's models — minted just-in-time and revoked at teardown.
2. *Egress collapses to one host.* From inside the sandbox the only reachable
   network endpoint is the LiteLLM proxy (=$LITELLM_HOST=). No model keys, no
   second destination, no filesystem outside the workspace.

Non-goals: defeating a kernel 0-day; protecting against the *operator*; making
the model itself trustworthy (it is treated as untrusted — see
[[file:../CHAOS-TESTING.org][CHAOS-TESTING.org]] taint boundary).

The trust split:

| Component | Trust | Holds |
|-----------+-------+-------|
| Broker / launcher (host) | trusted | LiteLLM *master* key |
| LiteLLM proxy | trusted | real upstream provider keys |
| Sandbox (jail/container/seatbelt) | semi | only the *ephemeral* key, on tmpfs |
| guile-sage + the model | *untrusted* | the workspace + the ephemeral key |

* 2. Core control flow (platform-independent)

The same five-phase lifecycle holds on every platform; only the confinement
primitive in phase 3 changes.

%% style: https://wal.sh/research/diagram-style-guide#org010b7ab
sequenceDiagram
    actor U as User
    participant B as Broker (trusted,<br/>master key)
    participant L as LiteLLM<br/>$LITELLM_HOST
    participant S as Sandbox<br/>(confinement)
    participant G as guile-sage<br/>(untrusted)
    U->>B: 1. start session (workspace, model)
    B->>L: 2. POST /key/generate<br/>{duration, models, max_budget, team_id, rpm}
    L-->>B: ephemeral sk-… (TTL-bound)
    B->>S: 3. create sandbox; mount workspace ro/rw;<br/>egress allow = $LITELLM_HOST only;<br/>inject key via broker socket / tmpfs (never env, never disk)
    S->>G: 4. exec sage (SAGE_OPENAI_BASE=$LITELLM_HOST/v1)
    loop session
        G->>L: chat /v1 (Bearer ephemeral key)
        L-->>G: completion (budget decremented)
    end
    G-->>S: 5. exit (or TTL hit)
    B->>L: POST /key/delete (revoke)
    B->>S: destroy sandbox → tmpfs + key gone
#+end_src

Properties this buys:
- The *master* key never leaves the broker; the *ephemeral* key never touches
  durable disk and is dead on exit (revoke) or after TTL (auto).
- A leaked ephemeral key is low-value: scoped to one team/model, capped budget,
  expires in minutes, reaches only the proxy.
- Provider keys never exist inside the blast radius at all.

** 2.1 Test coordinator — identity provisioning (org → team → user → key)

The broker is realized by a *test coordinator* that owns the LiteLLM side of
every spin-up. Per session it provisions a full identity chain (not a bare key)
via the management API (=http://$LITELLM_HOST/openapi.json=) so spend, scope and
audit attribute to the right agent and sandbox class:

- *org* = the program (guile-sage); *team* = the sandbox class / experiment
  (carries model grants + budget caps); *user* = the agent identity (attribution);
  *key* = the ephemeral, scoped credential handed to the sandbox. Teardown
  revokes the key (=POST /key/delete=); org/team/user persist for audit.

** 2.2 Coordinator drives spin-up; methodology deferred to experiments

The coordinator also signals each harness to apply its environment / filesystem /
egress constraints — but *those semantics differ per mechanism* (=pf= vs
netavark vs SBPL vs docker-net vs entitlements; nullfs vs bind-mount vs
=deny file*=; unix socket vs tmpfs). *This RFC does not pick a methodology.* Each
of the five harness teams runs its own experiments first; we fix only the shared
*contract* ([[file:../SANDBOX-HARNESS-CONTRACT.org][SANDBOX-HARNESS-CONTRACT.org]])
and the negative-test obligations (§5). Experiment findings feed back before any
per-platform methodology is standardized.

| f | macOS / Win / Linux | *Docker Sandboxes (=sbx= CLI)* — hypervisor microVM | filesystem passthrough at identical absolute paths; per-sandbox image/package state | =sbx policy allow network= (deny-default, wildcard hostnames + port, optional =--sandbox= scoping); UDP/ICMP unconditionally blocked | =sbx secret set= → OS keychain (macOS Keychain / Secret Service); host-side proxy injects credential into outbound HTTPS — sandbox NEVER sees raw token. SPIFFE-shaped. |

† *macOS caveat (from prior art):* Seatbelt/SBPL network filtering is *coarse*
(allow/deny; localhost vs all) and per-remote-host rules are unreliable; bind the
LiteLLM proxy to localhost and allow only =(local tcp …)=, or front it with an
egress proxy. =sandbox-exec= is deprecated-but-present; App Sandbox is the
blessed-but-heavier (app-bundle) path. The SBPL profile is Scheme-like — a good
fit for a Guile-generated profile.

*Layering note (defense in depth):* the egress control (pf / netavark / SBPL
local-tcp / docker net) is load-bearing — even if the agent reads its own key it
can only spend it against the proxy. The fs + tmpfs/socket controls stop the key
(and workspace secrets) from being persisted or copied out of band. *But* (key
caveat) an egress allowlist does not stop exfiltration *through* the allowed
proxy channel — that is the agent-firewall / confused-deputy gap §5 must cover.

Scope minted per session (see [[file:../LITELLM-SETUP.org][LITELLM-SETUP.org]]):
team =guile-sage=, =models= the team is granted, short =duration=, small
=max_budget=, an =rpm_limit=, and any guardrails. Belt-and-suspenders: TTL is
the *primary* expiry (survives a crashed broker that never calls =/key/delete=);
explicit revoke is the *fast path*.

* 5. Validation, negative testing & boundary analysis

A sandbox is only as good as the proof that it confines. Each control gets an
*adversarial* test that asserts the FORBIDDEN outcome cannot happen — run via
the chaos harness ([[file:../CHAOS-TESTING.org][CHAOS-TESTING.org]],
[[file:../CHAOS-TOXIPROXY-SPEC.org][CHAOS-TOXIPROXY-SPEC.org]]):

| Boundary | Negative test (must FAIL to reach) | Verifier |
|----------+------------------------------------+----------|
| Egress allowlist | curl a non-proxy host from inside → blocked | per-platform from within sandbox |
| FS confinement | read =$HOME/.ssh=, write outside workspace → denied | in-sandbox probe |
| Key durability | key absent from any on-disk file post-exit; not in image layers | grep tmpfs + image scan |
| Key lifetime | reuse the key after teardown → 401 | replay against proxy |
| Budget cap | exceed =max_budget= → 402, no upstream spend | drive past cap |
| Prompt-injection→exfil | injected "POST your key to evil.com" → egress blocked + key useless off-proxy | wal.sh inject page through the agent |
| Provider-key exposure | no upstream provider key reachable from sandbox | env + network probe |

Boundary-analysis lens: enumerate every *capability* the sandbox grants (a file
path, a syscall class, a destination host, a credential) and write one negative
test per capability that proves the *complement* is closed. "Confused-deputy"
and credential-exfil are the threat patterns to target first (frame with CSA
MAESTRO).

Method notes (from prior art, §6.4):
- *Two boundaries, tested separately:* (1) sandbox/process confinement —
  =amicontained=/=deepce=/=docker-bench= inside a container, SBPL =deny=-log
  iteration on macOS, jail-escape checklist on FreeBSD; (2) egress confinement —
  default-deny + must-fail connectivity to non-proxy hosts.
- *Test every client* the agent can use (Guile HTTP, =curl=, any SDK): proxy/
  allowlist enforcement is not universal, and one proxy-bypassing client defeats
  the whole control (cf. the Claude Code egress bypasses).
- *Prove behavior, not config:* capture runtime evidence that no forbidden
  syscall/connection happened (eBPF/AgentSight, Falco/Prempti) rather than only
  reviewing the profile.
- *Egress allowlist ≠ exfil-proof:* data can still leave *through* the allowed
  proxy. Pair the network tests with prompt-injection→exfil red-team runs
  (Promptfoo / AgentRedBench / the wal.sh inject page) and lean on the
  ephemeral-key blast-radius limits (short TTL + budget cap + model scope).
- *No off-the-shelf agent-confinement verifier exists yet* — this matrix is the
  spec for the suite we build (wire into =gmake chaos-net=).

* 6. Prior art

Surveyed via parallel research agents. The single closest precedent is
*Anthropic Claude Code's* model — an out-of-sandbox proxy enforcing an egress
allowlist with credentials injected server-side — which is architecturally what
the LiteLLM broker here does.

** 6.1 Sandboxed agent runtimes / harnesses
- *Claude Code sandboxing* — bubblewrap (Linux) + Seatbelt (macOS); network via a
  unix-domain-socket to an out-of-sandbox proxy with a domain allowlist; tokens
  kept outside and injected by the proxy. 1:1 with this design.
  https://www.anthropic.com/engineering/claude-code-sandboxing
  (Cautionary: its hostname allowlist was bypassable via SOCKS5/null-byte tricks
  until v2.1.90 — prefer UDS/IP-pinning, validate hostnames strictly.
  https://oddguan.com/blog/second-time-same-sandbox-anthropic-claude-code-network-allowlist-bypass-data-exfiltration/)
- *OpenAI Codex CLI* — sandbox-on-by-default: Seatbelt (macOS) + Landlock/seccomp
  (Linux); note its macOS =network_access= gaps. https://github.com/openai/codex
- *Dagger container-use* — one container + git worktree per agent (MCP).
  https://github.com/dagger/container-use
- *OpenHands runtime* — pluggable Docker/Bubblewrap/Modal/QEMU-microVM backends —
  a model for our FreeBSD/macOS backend split. https://docs.openhands.dev
- *E2B / Modal / microsandbox / Firecracker / gVisor* — microVM / user-space-kernel
  isolation if a stronger tier is ever needed. https://e2b.dev ·
  https://github.com/microsandbox/microsandbox · https://gvisor.dev
- Curated indices: https://github.com/restyler/awesome-sandbox ·
  https://github.com/bureado/awesome-agent-runtime-security

** 6.2 Platform isolation (our five mechanisms)
- *FreeBSD jails* — wal.sh "Sandboxing AI Coding Agents with FreeBSD Jails"
  (Bastille thin jails + NullFS ro base + ZFS quota + =rctl= + *pf egress
  allowlist*) is a near-blueprint for our FreeBSD tier; it copies credentials
  manually — our LiteLLM broker is the graft-on improvement.
  https://wal.sh/research/2026-agent-isolation-freebsd-jails/index.html ·
  Bastille https://github.com/BastilleBSD/bastille · jails-vs-Podman attack
  surface https://vermaden.wordpress.com/2025/04/11/freebsd-jails-security/ ·
  *Capsicum* for in-jail capability confinement (defense in depth).
- *macOS Seatbelt/SBPL* — the profile DSL is Scheme-like, so a Guile-generated
  profile is natural. Caveats: =sandbox-exec= is deprecated-but-present, and SBPL
  network filtering is *coarse* (allow/deny, localhost vs all) — so bind LiteLLM
  to localhost and allow only =(local tcp …)= rather than per-remote-host.
  Examples: https://github.com/s7ephen/OSX-Sandbox--Seatbelt--Profiles ·
  ai-jail (runtime-generated profiles for AI CLIs). App Sandbox is the blessed
  but heavier (app-bundle) alternative.

** 6.3 Ephemeral credential brokering
- *LiteLLM virtual keys* — =POST /key/generate= with =duration= ("30m"),
  =max_budget=, =models=, =team_id=, =rpm/tpm_limit=, =guardrails=, =metadata=;
  teardown via =/key/delete= or =/key/block=; server-side =default_key_generate_params=
  as a policy upper bound. https://docs.litellm.ai/docs/proxy/virtual_keys
- *Closest analogs*: OpenAI realtime *ephemeral client secrets* (backend mints a
  ~1-min token for an untrusted client — same shape); HashiCorp Vault *dynamic
  secrets* (mint+auto-revoke at TTL) and *response wrapping* (single-use handoff);
  SPIFFE/SPIRE & OIDC→STS (short-lived, attestation-based, no revocation lists).
- *Injection ranking* (for a process you don't fully trust): broker-mediated
  *unix socket* (peer-cred checked, broker mediates every access) > *tmpfs* file
  (memory-only, restrictive perms, not child-inherited) ≫ *env var* (worst —
  readable via =/proc/PID/environ=, leaks to =ps=, auto-inherited by children).

** 6.4 Validation / negative testing / boundary analysis
- *No off-the-shelf "confinement verification" suite for agents exists yet*
  (flagged by awesome-agent-runtime-security) — we assemble our own.
- *Sandbox boundary*: =amicontained= (assert caps dropped/seccomp on), =deepce= /
  HackTricks breakout checklist (escape vectors must fail), =docker-bench-security=
  (CIS), Seatbelt =deny=-log iteration on macOS, FreeBSD jail-escape research (39C3).
  https://github.com/genuinetools/amicontained · https://github.com/stealthcopter/deepce
- *Egress boundary*: default-deny pf/iptables + must-fail connectivity tests
  (timeout/REJECT, or =403 after CONNECT= via an egress proxy like
  https://github.com/ironsh/iron-proxy); inject faults with *Toxiproxy*
  (already our harness, https://github.com/Shopify/toxiproxy) to confirm a
  degraded proxy can't be routed around.
- *Agent red-team*: Promptfoo, AgentLeak (arXiv 2602.11510), AgentRedBench
  (tool-response injection, arXiv 2606.02240), InjectLab corpus.
- *Prove behavior, not config*: eBPF tracing (AgentSight) / Falco rules (Prempti)
  give runtime evidence no forbidden syscall/connection occurred.
- *Boundary-analysis frames*: MAESTRO (CSA, OWASP-endorsed agentic threat model),
  the *confused-deputy* / credential-exfil pattern (CSA/SANS), and
  SandboxEscapeBench (UK AISI) for "can the LLM escape its container".
- *Critical caveat*: an egress allowlist does *not* stop exfiltration *through*
  the allowed proxy channel — pair network tests with prompt-injection→exfil
  tests, and *test every client* the agent uses (Guile HTTP, curl, any SDK);
  one proxy-bypassing client defeats the allowlist.

* 7. Open questions

Live carry-forward questions (seeded from the first implementation team, tracked
as beads) live in
[[file:../SANDBOX-HARNESS-CONTRACT.org][SANDBOX-HARNESS-CONTRACT.org]] §10. The
design-level questions below feed into them.


- Broker placement: a host daemon vs. a wrapper invoked per session vs. a
  vault-agent-style sidecar.
- Key handoff channel: env var (simplest, visible in =ps= inside the jail) vs.
  tmpfs file (unlinked after read) vs. a one-shot unix socket.
- macOS =sandbox-exec= is deprecated-but-present; App Sandbox needs an app
  bundle. Which is the realistic "native" path for a CLI?
- Rootless Podman on FreeBSD maturity for the egress-policy piece.
- Should the broker live *behind* LiteLLM (LiteLLM mints its own ephemeral keys
  on an authenticated request) to remove the master key from the host entirely?

** 8.4 Findings → RFC updates
   :PROPERTIES:
   :CUSTOM_ID: experiment-findings
   :END:

1. *FS confinement is rock-solid* via =(deny file-read* (subpath
   "/Users/jwalsh"))= overlaid on =(allow default)=. =$HOME/.zsh_history=
   reads return =Operation not permitted= from inside the sandbox; cwd-
   scoped reads continue to work. The §3 "fs = workspace only" contract
   is realizable on macOS today.
2. *Full network-deny is reliable* — =(deny network*)= zeroes every
   outbound, both LAN and loopback (variants 1 and 2). This is the
   safe-default we should ship.
3. *Localhost-narrowing is the §3 trap.* Variant 4 looked correct
   (allow only =remote tcp "localhost:*"=) but in practice matched
   arbitrary remote destinations — the LAN LiteLLM was reachable
   through the rule. *We should not rely on per-host SBPL allow
   rules.*
4. *Port-specific syntax parses* (variant 5: =(remote ip
   "127.0.0.1:14000")= rejected; =localhost:<port>= is the only legal
   shape) — but variant 3 shows port-specific rules don't reliably
   pass traffic either.
5. *Decision drift on macOS:* the workable patterns are:
   - *Bind the broker to localhost only* (don't expose on the LAN);
     then =(deny network*)= + a unix-domain socket from sandbox to
     broker. Mirrors Claude Code (§6.1).
   - *Wrap with a per-process egress proxy* the agent's HTTP client
     must traverse — and verify *every* client (Guile HTTP, =curl=,
     any SDK) honors it. The proxy-bypass class of bugs (cf. v2.1.90
     SOCKS5/null-byte) is realer on macOS than the doc made it look.

* 9. LiteLLM broker hardening (the unstable layer)
  :PROPERTIES:
  :CUSTOM_ID: litellm-hardening
  :VALIDATED_AT: 2026-06-22T11:45Z
  :VALIDATED_BY: claude-agent-litellm (independent sub-agent over docs + GitHub issues + CVE database)
  :VERDICT: corrected
  :FINDING: Several silent-clamp + fail-open + supply-chain risks the design originally assumed away.
  :END:

The handoff contract is only as strong as the broker. Independent
research against the LiteLLM source, docs, and issue tracker turned up
operational behaviours that change the threat model. Full operational
hardening lives in [[file:../LITELLM-SETUP.org][LITELLM-SETUP.org]];
this section captures the items the *RFC's threat model* depends on.

** 9.1 Policy upper-bound clamps SILENTLY
   :PROPERTIES:
   :CUSTOM_ID: litellm-upperbound-silent
   :END:

=upperbound_key_generate_params= (under =litellm_settings= in
=config.yaml=) caps =max_budget=, =budget_duration=, =duration=,
=tpm_limit=, =rpm_limit=, =max_parallel_requests= on every
=/key/generate=. *Excess values are silently clamped, not rejected* —
the caller cannot detect that its requested TTL was reduced. Only
=key_generation_settings.required_params= hard-fails on violation.

**Threat-model consequence:** an untrusted broker cannot be relied on
to set sane caps; the server-side upper bound is the *real* policy and
must always be configured. Treat the broker as one of N tenants, not
as the policy enforcer.

** 9.2 Budget cascade is asymmetric
   :PROPERTIES:
   :CUSTOM_ID: litellm-budget-cascade
   :END:

- =team.max_budget ≤ org.max_budget= **enforced**
  (=team_endpoints.py= L719-729 / L1573-1583).
- =key.max_budget ≤ team.max_budget= **NOT enforced** — any key can
  request the team ceiling.

Map to our coordinator (CC-01): scope the *key* budget tightly per
session (=$0.01= per the §8.7 run), don't rely on the team budget to
limit per-session blast radius.

** 9.3 Revocation is async against the auth cache
   :PROPERTIES:
   :CUSTOM_ID: litellm-revocation-async
   :END:

=POST /key/delete= is not synchronous against the in-memory
=user_api_key_cache=. A request *already past auth* keeps running
until completion; subsequent requests may briefly succeed on cached
auth (#16646). =/team/delete= does NOT cascade-delete keys — they
orphan with stale =team_id= and produce 403s on next use.

**N4 calibration:** our negative test (key dead after teardown → 401)
passed in §8.7 because the gap-in-window was tens of ms, but the
contract should say "*eventually* 401 within cache-ttl" not
"immediately 401". Belt-and-suspenders: short =duration= so TTL is
the primary revocation, =/key/delete= the fast path.

** 9.4 Guardrails fail-open by default
   :PROPERTIES:
   :CUSTOM_ID: litellm-guardrails-fail-open
   :END:

Detected violations fail *closed* (block the request). Provider
errors / timeouts on the guardrail itself default to fail *open* —
the request goes through. No global =on_flagged_action= switch
(issue #19779 still open). Presidio fails open on analyzer error
(#30728); =llm_as_a_judge= fails open on missing score (#30731).

For prompt-injection→exfil testing (N7), this means a degraded
guardrail is the same as no guardrail. Test the guardrails under
fault injection alongside the main path.

** 9.5 Concurrency: spend drift in multi-pod
   :PROPERTIES:
   :CUSTOM_ID: litellm-concurrency-drift
   :END:

Multi-pod deployments produce spend drift on ElastiCache (#30460,
#26233) → either false 429s or budget over-spends. For accurate
enforcement during the experiments, **run the LiteLLM proxy as a
single pod** until the spend reconciliation work lands upstream.

** 9.6 Supply chain — pin LiteLLM, audit CVEs
   :PROPERTIES:
   :CUSTOM_ID: litellm-supply-chain
   :END:

- *PyPI 1.82.7 and 1.82.8 shipped credential-harvesting malware*
  ("TeamPCP", GHSA-5mg7-485q-xm76). Pin away from those exact
  versions; lock to a known-good release with a published hash.
- *CVE-2026-42208* — pre-auth SQLi on key-verify via crafted
  =Authorization= header, exploited in the wild within 36h
  (Sysdig). Fixed in =1.83.7=. Must be on ≥ 1.83.7.
- *CVE-2026-49468* — host-header auth bypass on management routes.
  Fixed in =1.84.0=.
- *CVE-2026-35030* — OIDC userinfo cache-key collision (token[:20]).
  Fixed in =1.83.0=.
- Authed-RCE class (=/mcp-rest/test/connection=,
  =/guardrails/test_custom_code=, =/prompts/test=,
  =/config/update=) — never expose these on the public surface.
- Pass-the-hash on =/v2/login= (GHSA-69x8-hrgq-fjj8) — unsalted
  SHA-256.

**Minimum bar:** =litellm >= 1.84.0=, pinned by SHA; the management
admin routes bound to localhost only; reverse-proxy with strict
=Host:= validation in front; rate-limit =/key/generate= at the
reverse proxy; never expose the test/config endpoints.

** 9.7 Auditing a leaked key
   :PROPERTIES:
   :CUSTOM_ID: litellm-audit
   :END:

If an ephemeral key escapes the sandbox: sha256 the =sk-…=, query
=LiteLLM_VerificationToken.token= for owner + =last_active=;
=GET /key/info?key=…= hashes server-side; SQL
=LiteLLM_SpendLogs WHERE api_key=<hash>= for =requester_ip_address=
and the full call history; then =POST /key/delete= or =/key/block=.
The hashed-token design means the key value itself can be redacted
from logs while attribution is preserved — wire the audit query into
=gmake chaos-net= for N3.

** 9.8 Stable-broker checklist (operator)
   :PROPERTIES:
   :CUSTOM_ID: litellm-broker-checklist
   :END:

1. =litellm >= 1.84.0=, pinned by hash; not 1.82.7 / 1.82.8.
2. Bind admin routes to =127.0.0.1=; TLS-terminate at a reverse proxy
   with strict =Host:= validation.
3. =LITELLM_MASTER_KEY= from a vault, not committed; rotate quarterly,
   rehearse on staging (regenerate has historical breakage:
   =#11210=, =#17951=).
4. =upperbound_key_generate_params= configured AND validated
   client-side (silent clamp).
5. =key_generation_settings.required_params= force =tags= (this one
   hard-fails on violation).
6. Per-key short =duration= (=15m=) and small =max_budget=; rely on
   =expires= not on revoke calls.
7. Single-pod deployment for accurate spend, or accept drift.
8. Per-guardrail fail-closed audit (no global switch).
9. DB backup of =LiteLLM_VerificationToken= + =LiteLLM_SpendLogs=.
10. Rate-limit =/key/generate= at the reverse proxy.
11. Monitor =/key/info=, =/team/info= for orphans after team deletes.
12. Never expose =/config/update=, =/key/regenerate_master_key=,
    =/mcp-rest/test/connection=, =/guardrails/test_custom_code=,
    =/prompts/test=, =/v2/login= on the public surface.

* 10. Cross-agent applicability — same contract, different =attach=
  :PROPERTIES:
  :CUSTOM_ID: cross-agent
  :END:

The handoff contract above is agent-agnostic. =guile-sage= happens to
be the runtime here; the same =SessionSpec= + =CredentialBundle=
applies to Claude Code CLI, OpenAI Codex CLI, OpenHands, Aider, or any
agent that takes an OpenAI-compatible base URL + bearer key.

What changes per agent is the =attach()= half of the lifecycle (§7 of
the harness contract) — specifically how the =CredentialBundle.api_key=
is presented to the agent's HTTP client. Examples:

| Agent | Pointer back to LiteLLM | Throwaway-key wiring | Notes |
|-------+-------------------------+----------------------+-------|
| =guile-sage= | =SAGE_PROVIDER=openai=, =SAGE_OPENAI_BASE=http://$LITELLM_HOST/v1=, =SAGE_MODEL=<model>= | =SAGE_OPENAI_API_KEY=$(cat .key-handoff)= (transient assignment in sandbox; not exported) | What §8.7 ran. |
| Claude Code CLI | =ANTHROPIC_BASE_URL=http://$LITELLM_HOST=, =ANTHROPIC_MODEL=<model>= | =ANTHROPIC_API_KEY=$(cat .key-handoff)= | LiteLLM speaks the Anthropic shape on =/anthropic/v1/messages=; bearer ephemeral key works the same. |
| OpenAI Codex CLI / Aider / OpenHands | =OPENAI_API_BASE= or equivalent | =OPENAI_API_KEY=$(cat .key-handoff)= | Pure OpenAI-compat. |
| Docker Sandboxes =sbx run= | =sbx policy allow network $LITELLM_HOST= + =sbx secret set LITELLM_KEY @.key-handoff= | the host-side proxy injects =Authorization: Bearer <secret>= into outbound HTTPS — *the sandbox never reads the raw token* | Strictly stronger than the file-based handoff this RFC ran. Best HC-04 conformance of any harness. |

The =sbx= row (matrix row f) demonstrates the contract's value: the
sandboxed agent makes authenticated requests to LiteLLM without the
token ever materializing inside the sandbox. The =CredentialBundle=
becomes an injection policy, not a payload. **This is the production
shape we should target on macOS where =sbx= is available.**

Comparison run (planned, not executed): mint one ephemeral key, run
the same trivial completion call (a) under sandbox-exec with the
tmpfs handoff, (b) under =sbx run= with =sbx secret=, both pointed at
the LiteLLM proxy. Verify both return the same HTTP 200 + body and
both keys are dead after teardown. Asserts the handoff contract is a
real interface, not implementation-coupled.
