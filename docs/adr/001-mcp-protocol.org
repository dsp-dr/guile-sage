#+TITLE: ADR-001: MCP Protocol Integration
#+DATE: 2026-01-28
#+STATUS: Rejected
#+DECIDERS: dsp-dr

* Context

Model Context Protocol (MCP) is Anthropic's protocol for connecting LLMs to external tools and data sources. It uses JSON-RPC 2.0 over stdio and has growing ecosystem adoption.

We evaluated MCP for guile-sage in two capacities:
1. *MCP Client*: sage calling external MCP servers
2. *MCP Server*: sage exposing its tools via MCP

* Decision

*We will NOT implement MCP protocol support.*

Instead, we will define a native Guile plugin contract (see RFC-001).

* Rationale

** MCP Client (calling external servers)

| Consideration | Assessment |
|---------------+------------|
| Complexity | High - JSON-RPC over stdio, subprocess management |
| Dependencies | Node.js runtime for most MCP servers |
| Value | Low - sage can shell out directly for same effect |
| Maintenance | High - protocol versioning, error handling |

For specific integrations (Cloudflare, Atlassian, etc.), building native Guile clients is simpler and more maintainable than wrapping MCP servers.

** MCP Server (exposing sage tools)

| Consideration | Assessment |
|---------------+------------|
| Use case | Unclear - if running sage, already have its tools |
| Complexity | Medium - implement JSON-RPC server |
| Value | Low - no identified consumers |

** What MCP Gets Right

MCP's value is as a *registry and specification*, not a runtime protocol:
- Well-defined tool schemas
- Standard capability descriptions
- Reference implementations

We can reference MCP's patterns without adopting the protocol.

* Alternatives Considered

** 1. Full MCP Implementation
- Pros: Ecosystem compatibility, standard protocol
- Cons: Complexity, Node.js dependency, unclear value
- *Rejected*

** 2. MCP Client Only
- Pros: Access to MCP server ecosystem
- Cons: Most servers are Node.js, can shell out instead
- *Rejected*

** 3. Native Plugin Contract (Selected)
- Pros: Simple, Guile-native, no dependencies, tailored to our needs
- Cons: Not ecosystem-compatible
- *Selected* - see RFC-001

** 4. No Plugin System
- Pros: Simplicity
- Cons: Limits extensibility
- *Rejected* - some extensibility needed

* Consequences

** Positive
- No JSON-RPC complexity
- No Node.js subprocess management
- Simpler codebase
- Plugin contract tailored to Guile

** Negative
- Cannot directly use MCP servers
- Not compatible with Claude Desktop tool ecosystem
- Must build integrations natively

** Neutral
- Can still reference MCP specs for tool design patterns
- Future reconsideration possible if ecosystem matures
