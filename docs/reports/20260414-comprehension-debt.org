#+TITLE: Comprehension Debt in AI-Assisted Codebases
#+SUBTITLE: Measurement, Mitigation, and Implications for Agentic Development

* Executive Summary

Comprehension debt is the growing gap between how much code exists in a
system and how much of it any human genuinely understands --- a
restatement, in AI-assisted terms, of Naur's "Programming as Theory
Building" (1985) thesis that the program is the theory in the
programmer's mind, not the artifact in the repository.  Unlike
technical debt, which announces itself through friction, comprehension
debt breeds false confidence: the codebase looks clean, tests are green,
velocity metrics hold steady --- and the reckoning arrives quietly.

This report surveys the emerging 2026 literature on comprehension debt,
applies its frameworks to the guile-sage codebase, and proposes concrete
eval dimensions and mitigations for agentic development workflows.

* Literature Survey

** Addy Osmani: "Comprehension Debt" (March 2026)

Source: [[https://addyosmani.com/blog/comprehension-debt/][AddyOsmani.com]]

*** Core thesis: Speed asymmetry

#+BEGIN_QUOTE
AI generates code far faster than humans can evaluate it.
#+END_QUOTE

Historically, senior engineers could review faster than junior engineers
could write.  AI inverts this: /a junior engineer can now generate code
faster than a senior engineer can critically audit it/, removing the
rate-limiting factor that kept review meaningful.

*** Tests: necessary but insufficient

#+BEGIN_QUOTE
A test suite capable of covering all observable behavior would, in many
cases, be more complex than the code it validates.
#+END_QUOTE

Osmani's argument: you cannot test for unanticipated edge cases.  Nobody
writes a test asserting that dragged items should not turn completely
transparent.  Green tests prove mechanical correctness, not systemic
comprehension.

*** Proposed mitigations

1. *Specification discipline*: write detailed specs before implementation
2. *System-level understanding*: maintain mental models enabling
   architectural-scale error detection
3. *Honest verification*: distinguish between "tests passed" and genuine
   comprehension
4. *Countervailing metrics*: time-to-root-cause, unassisted debugging
   success rate, onboarding depth, design decision documentation

** Osmani: "The 80% Problem in Agentic Coding" (April 2026)

Source: [[https://addyo.substack.com/p/the-80-problem-in-agentic-coding][Substack]]

The threshold has shifted from 70% to 80%+ of code generated by AI
agents.  Andrej Karpathy described the inversion: "I rapidly went from
about 80% manual coding and 20% agents to 80% agent coding."

Key data points:

| Metric | Value |
|--------|-------|
| Developers reporting 80% problem | 66% |
| Saying debugging AI code takes longer than writing it | 45% |
| PR volume increase (high-adoption teams) | +98% |
| Review time increase | +91% |
| PR size increase | +154% |
| Developers who consistently verify before committing | 48% |

The verification paradox: 38% of developers find reviewing AI-generated
logic requires /more/ effort than reviewing human-written code, despite
its surface cleanliness.

Successful practitioners spend 70% of effort on problem definition and
verification strategy, 30% on execution --- inverting traditional ratios.

** Anthropic: "How AI Impacts Skill Formation" (January 2026)

Source: [[https://www.anthropic.com/research/AI-assistance-coding-skills][Anthropic Research]]

*** Study design

A randomized controlled trial with 52 mostly junior software engineers
learning Trio (Python async library).  Both groups completed coding tasks
followed by a comprehension quiz.

*** Key findings

| Measure | AI-assisted | Hand-coding | Delta |
|---------|-------------|-------------|-------|
| Quiz score | 50% | 67% | -17pp |
| Cohen's d | | | 0.738 |
| p-value | | | 0.01 |
| Speed gain | +2 min | baseline | not significant |

Largest declines occurred in *debugging*, with smaller but significant
drops in conceptual understanding and code reading.

*** The critical distinction: passive delegation vs. active inquiry

Developers who scored 65%+ used AI for /conceptual questions/ --- asking
"how?" and "why?" while coding independently.  Developers who scored
below 40% delegated code generation to AI without interrogation.

Three high-retention patterns:
1. *Generation-then-comprehension*: generate code, then ask follow-up questions
2. *Hybrid code-explanation*: request code and explanations simultaneously
3. *Conceptual inquiry*: ask only conceptual questions, resolve errors independently

** Margaret-Anne Storey: Cognitive Debt (February 2026)

Sources: [[https://margaretstorey.com/blog/2026/02/09/cognitive-debt/][Original post]] and [[https://margaretstorey.com/blog/2026/02/18/cognitive-debt-revisited/][Follow-up]]

*** The "theory of the system"

Drawing on Peter Naur's foundational work, Storey emphasizes that "a
program is a theory that lives in the minds of the developer(s)."  This
theory captures what the program does, how intentions manifest in code,
and how future modifications can be made.

#+BEGIN_QUOTE
Technical debt lives in the code.  Cognitive debt lives in people.
#+END_QUOTE

Cognitive debt: the accumulated gap between a system's evolving structure
and a team's shared understanding of how and why that system works.

*** Student team study

In an entrepreneurship course, student teams building software products
hit a critical breakdown by weeks 7-8.  The team initially blamed
technical debt but discovered the real problem: "no one on the team could
explain why certain design decisions had been made or how different parts
of the system were supposed to work together."

Warning signs:
- Team hesitation to make changes (fear of ripple effects)
- Reliance on "tribal knowledge" held by one or two individuals
- Systems perceived as "black boxes"
- Silent loss of shared theory

*** Where system knowledge lives

The theory is distributed across: people, documentation, tests,
conversations, tooling, and increasingly, AI agents --- requiring
holistic maintenance beyond code refactoring.

** Epistemic Debt: Metacognitive Scripts (arXiv 2602.20206)

Source: [[https://arxiv.org/html/2602.20206v2][arXiv]]

A between-subjects experiment (N=78) using a Cursor IDE plugin defined
epistemic debt formally:

#+BEGIN_QUOTE
Ed(t) = Uf(t) - Cc(t)
#+END_QUOTE

where functional utility minus corrective competence equals accumulated
debt.

*** Quantified results

| Condition | Utility | Repair success | Epistemic debt |
|-----------|---------|----------------|----------------|
| Unrestricted AI | 92.4% | 23.1% | 69.3 points |
| Scaffolded AI | 89.1% | 61.5% | 27.6 points |
| Manual control | 65.0% | 69.2% | -4.0 points |

The "Explanation Gate" --- requiring novices to articulate causal logic
before integrating AI-generated code --- reduced epistemic debt by 60%
(69.3 to 27.6) with only a 3.3 percentage-point loss in functional
utility.

64% of scaffolded learners later acknowledged the gate forced recognition
of knowledge gaps they would otherwise have missed.

** Triple Debt Model (arXiv 2603.22106)

Source: [[https://arxiv.org/abs/2603.22106][arXiv]]

Proposes three interacting debt types:

| Debt type | Where it lives | What erodes |
|-----------|---------------|-------------|
| Technical | In the code | Design quality, maintainability |
| Cognitive | In people | Shared mental models, team reasoning |
| Intent | In externalized knowledge | Design rationale, goals, constraints |

Intent debt is distinct from cognitive debt: it concerns /missing
externalized knowledge/ rather than /eroded team understanding/.  AI
agents need externalized intent as much as humans do.

** Andrej Karpathy: The "Yoink" Thesis

Sources: [[https://github.com/theogbrand/yoink][GitHub]], [[https://cybernews.com/ai-news/andrej-karpathy-slopacolypse/][Cybernews]]

Karpathy's position: use LLMs to "yoink" (reimplement) dependency
functionality rather than pulling in opaque third-party packages.  Write
code like bacterial genomes: small, modular, self-contained.

The comprehension argument: agents reason better from reimplemented
functionality they have full visibility into, over opaque third-party
libraries.  It was sometimes cheaper to have the agent reimplement
subsets of functionality than to work around opaque upstream behavior.

This is directly relevant to comprehension debt: every dependency you
pull in is code nobody on your team understands.  Yoinking is a
comprehension-preserving strategy.

Karpathy's prediction for 2026: "the year of the slopacolypse across all
of GitHub" --- a flood of AI-generated code that overwhelms traditional
verification.

** Dan Abramov: "The WET Codebase" (Deconstruct 2019)

Source: [[https://overreacted.io/the-wet-codebase/][Overreacted]]

Abramov's argument: strict adherence to DRY (Don't Repeat Yourself)
inevitably leads to software we cannot understand.  Premature abstraction
creates accidental coupling and opaque layers.

Relevance to comprehension debt: AI is the ultimate DRY machine.  It
abstracts aggressively, collapses patterns, and eliminates repetition ---
often eliminating the very redundancy that aided human understanding.
Allowing some duplication preserves comprehensibility.

** Charity Majors: Observability and System Understanding

Sources: [[https://charity.wtf/category/observability/][charity.wtf]], [[https://www.infoq.com/articles/charity-majors-observability-failure/][InfoQ]]

Majors' key insight: you cannot understand a system by reading its code
alone.  You must observe it in production.

#+BEGIN_QUOTE
Being able to visualize your work is what makes engineering abstractions
accessible to actual engineers.  You shouldn't have to be a world-class
engineer just to debug your own damn code.
#+END_QUOTE

Short deployment cycles directly enable understanding: when cycles are
measured in minutes, engineers maintain mental models of changes.
Batching obscures causation.

For comprehension debt: observability is the runtime counterpart of code
comprehension.  A team that cannot trace a request through its system has
comprehension debt in production, regardless of how well they understand
the source.

** Jay Wengrow: "A Common-Sense Guide to AI Engineering" (May 2026)

Source: [[https://pragprog.com/titles/jwpaieng/a-common-sense-guide-to-ai-engineering/][Pragmatic Programmers]]

Wengrow's book covers building production-ready LLM applications with
chapters on automated evals and guardrails.  Key principles:

- *Evals*: automated checks that keep constant tabs on application
  quality --- not just correctness, but behavioral alignment with intent.
- *Guardrails*: protective mechanisms that filter content and constrain
  behavior, separate from the model itself.

The eval/guardrail distinction maps to the comprehension debt question:
evals measure /what the system does/; guardrails constrain /what it
should not do/.  Neither measures /whether the team understands why/.

** Armin Ronacher: "The Final Bottleneck" (February 2026)

Source: [[https://lucumr.pocoo.org/2026/2/13/the-final-bottleneck/][lucumr.pocoo.org]]

Ronacher argues from queuing theory: when input (AI-generated code) grows
faster than throughput (human review), the system accumulates failures.
He cites OpenClaw's 2,500+ open pull requests as evidence.

#+BEGIN_QUOTE
Machines cannot be accountable.
#+END_QUOTE

Human comprehension becomes irreplaceable because accountability requires
understanding.  If machines generate code and machines review it, humans
become rubber-stampers --- and when the system fails, nobody can explain
why.

** Derick Chen: The Generation-Verification Asymmetry

Source: [[https://www.buildwithdc.co/posts/the-new-asymmetry-when-generation-outpaces-verification-in-ai-native-development/][buildwithdc.co]]

Chen identifies the paradox: component-level code quality improves while
system-level quality degrades, creating what he terms the "big ball of
slop."  Proposed solution: use a separate LLM instance with intentionally
different context to verify artifacts --- not the same agent checking its
own work.

** Michelle Pellon: "The AI Risk Nobody's Measuring"

Source: [[https://michellepellon.com/blog/2026-02-23-cognitive-debt][michellepellon.com]]

Proposed measurable proxies for cognitive debt:

| Proxy metric | Source | Finding |
|-------------|--------|---------|
| Refactoring % of code changes | GitClear | Declined from 25% to <10% (2021-2024) |
| Security vuln density in AI code | Research | 1.7x more major issues, 2.74x more security vulns |
| Cognitive offloading correlation | Gerlich | r=+0.72 with AI use, r=-0.75 with critical thinking |
| Actual vs perceived productivity | METR | 19% actual slowdown despite 20% perceived speedup |

Recommended practice layers:
- *Layer 1 (Culture)*: Budget 20-30% extra time for comprehension
- *Layer 2 (Lightweight)*: Decision logs, ownership reviews
- *Layer 3 (High-Stakes)*: Design intent before generation, structured
  comprehension loops, monthly theory rebuilds

* The Speed Asymmetry Problem

** Quantifying the gap

| Dimension | AI speed | Human speed | Ratio |
|-----------|----------|-------------|-------|
| Code generation | 140-200 lines/min | 20-40 lines/min (writing) | 5-7x |
| Review capacity | N/A | Same as always | bottleneck |
| PR volume (2026) | +29% YoY | flat | diverging |

The 5-7x generation gap is compounded by the fact that /reviewing/
AI-generated code takes more cognitive effort than reviewing human code
(38% of developers report this), because it lacks the incremental
reasoning trail that human-written code carries implicitly.

** The review bottleneck

The traditional software development pipeline assumed generation was the
bottleneck.  AI removed that constraint and revealed the true bottleneck:
human comprehension.  This is analogous to the British Industrial
Revolution pattern Ronacher cites --- removing one constraint (spinning)
revealed the next (weaving).

In the agentic coding context: the junior developer (or agent) generates
faster than the senior can audit.  PR counts go up.  Review depth goes
down.  Comprehension debt accumulates.

* Tests vs. Understanding: When Green Is Not Enough

** The coverage illusion

Green tests create a mechanical proof of /specified behavior/ but not of:

1. *Unspecified behavior*: edge cases nobody thought to test
2. *Systemic properties*: how modules interact under load, in sequence,
   or with unexpected input combinations
3. *Design intent*: why the code is structured this way, not just that
   it works
4. *Negative space*: what the code intentionally does NOT do

** The epistemic debt formula applied

From the metacognitive scripts paper: Ed(t) = Uf(t) - Cc(t).

Unrestricted AI achieves 92.4% functional utility (tests pass) but only
23.1% corrective competence (ability to fix bugs).  Epistemic debt:
69.3 points.  This is the formal expression of "green tests, no
understanding."

** When are green tests actually sufficient?

Tests suffice when the following conditions hold simultaneously:

1. The person who wrote the tests understands the full spec (not just
   happy paths)
2. The tests encode design intent, not just input-output pairs
3. Someone on the team can explain /why/ each test exists
4. The test suite is maintained by people who understand the code
   (not just the AI that generated both)

When AI generates both code and tests, the epistemic debt formula applies
to /both artifacts/: the tests themselves become part of the
comprehension gap.

* Application to guile-sage

** Codebase profile

| Metric | Value |
|--------|-------|
| Total source modules | 21 |
| Total source LOC | 8,343 |
| Total test LOC | 6,356 |
| Test-to-source ratio | 0.76 |
| Test suites | 24 |
| Total commits | 300 |
| AI co-authored commits | 250 (83%) |
| Architecture Decision Records | 5 |
| Eval test cases | 20 (traffic) + 4 (algorithmic) |

** Comprehension risk indicators

*** High AI authorship ratio

250 of 300 commits (83%) are AI co-authored.  This is not inherently
problematic --- but it means the comprehension preservation burden falls
on the /review phase/, not the /writing phase/.  If review becomes
rubber-stamping, the entire codebase becomes a black box.

*** Untested modules

Ten modules (3,362 lines, 40% of codebase) lack dedicated test files:

| Module | Lines | Role | Risk |
|--------|-------|------|------|
| util.scm | 680 | HTTP/JSON utilities | Medium (foundational) |
| logging.scm | 634 | Structured JSONL logging | Low (diagnostic) |
| gemini.scm | 452 | Gemini client | High (provider) |
| openai.scm | 423 | OpenAI-compat client | High (provider) |
| config.scm | 368 | XDG, env vars, .env | Medium (cross-cutting) |
| agent.scm | 283 | In-memory task system | Medium |
| irc.scm | 275 | IRC integration | Low (optional) |
| status.scm | 106 | Display helpers | Low |
| main.scm | 104 | CLI entry point | Low |
| version.scm | 37 | Semver constants | Low |

The two untested provider modules (gemini.scm, openai.scm) together
represent 875 lines of network protocol code.  This is a /comprehension/
gap, not just a /coverage/ gap: if the only verification was AI-generated
visual inspection during development, no human may deeply understand
their error handling paths.

*** Module dependency complexity

The REPL module (repl.scm, 1,112 lines) imports 15 other sage modules
--- it is the hub of the entire system.  Its comment density is the
lowest in the codebase at 9%.  This module is the primary comprehension
debt risk: it is the largest, most connected, and least documented.

Dependency fan-in (modules that import a given module):

| Module | Imported by |
|--------|------------|
| config | 15 modules (every other module) |
| util | 7 modules |
| logging | 8 modules |
| tools | 2 modules (repl, mcp) |
| repl | 1 module (main) |

*** Comment density distribution

| Range | Modules | Examples |
|-------|---------|---------|
| >30% | 3 | commands (33%), model-tier (34%), provider (35%) |
| 20-30% | 7 | compaction, config, context, ollama, status, telemetry, version |
| 10-20% | 9 | agent, gemini, irc, mcp, openai, repl, session, tools, util |
| <10% | 2 | repl (9%), main (12%) |

The modules most likely to accumulate comprehension debt (large, complex,
frequently modified) tend to have the /lowest/ comment density.

** Does the eval harness test comprehension?

The current eval harness has two components:

1. *Algorithmic problems* (problems.scm): 4 problems testing code
   generation quality.  Rubrics include correctness, edge-case handling,
   and code quality --- but /not/ whether the model can explain its
   approach.

2. *Traffic eval cases* (traffic-cases.json): 20 cases across 5
   categories (tool_call, chain, planning, synthesis, guardrail).  Only
   one case (TC13: "Which source module is the largest and why might that
   be?") approaches a comprehension question.  No case asks the model to
   explain /why/ it chose a particular tool or /what trade-offs/ it
   considered.

The eval harness tests /correct tool call/ but not /understanding/.

* Proposed Eval Dimensions for Comprehension

** The missing dimension: "why?"

Current eval categories and their comprehension coverage:

| Category | Tests "what" | Tests "how" | Tests "why" |
|----------|-------------|-------------|-------------|
| tool_call | Yes | No | No |
| chain | Yes | Partially | No |
| planning | Partially | Partially | No |
| synthesis | Yes | Partially | No |
| guardrail | Yes | No | No |

A comprehension-aware eval should add:

*** 1. Rationale probes

Prompt: "List the files in src/sage/, then explain which module you would
modify to add a new compaction strategy and /why/ that module is the
right place."

Scoring: Does the response demonstrate understanding of the module
decomposition principle, or does it pattern-match on file names?

*** 2. Counterfactual questions

Prompt: "Why does repl.scm import both ollama.scm and provider.scm? What
would break if we removed the direct ollama import?"

Scoring: Does the response identify that ollama.scm provides streaming
primitives used directly by the REPL, while provider.scm provides the
dispatch abstraction?

*** 3. Dependency tracing

Prompt: "Trace the data flow from a user typing a message in the REPL to
the telemetry counter being emitted."

Scoring: Can the model name the correct module sequence
(repl -> provider -> ollama/openai/gemini -> telemetry) with accurate
descriptions of what each module contributes?

*** 4. Design trade-off articulation

Prompt: "The tool registry splits tools into safe and unsafe.  What is
the security model, and what are its limitations?"

Scoring: Does the response identify the YOLO-mode bypass, the
workspace-scoping mechanism, and the /tmp traversal risk that was
recently patched?

*** 5. Failure mode prediction

Prompt: "If the Ollama server becomes unreachable mid-conversation, what
happens to the session state? Where would you look to debug this?"

Scoring: Does the response correctly identify session persistence
behavior, error handling in ollama.scm, and the logging module's role?

** Comprehension quiz as eval gate

Inspired by the epistemic debt paper's "Explanation Gate": after the
model generates code or a tool-call plan, require it to explain /why/
before execution.  Score the explanation using SOLO taxonomy rubrics
(as in the arXiv 2602.20206 study):

| Level | Description | Score |
|-------|-------------|-------|
| Pre-structural | No relevant understanding | 1 |
| Uni-structural | One relevant aspect | 2 |
| Multi-structural | Several relevant aspects | 3 |
| Relational | Integrated understanding | 4 |
| Extended abstract | Generalizes beyond given | 5 |

Gate threshold: >= 3 (Relational) before proceeding.

* The 3-Phase Cycle and Comprehension Debt

** Develop / Review / Harden

The guile-sage development cycle (implicit in CLAUDE.md):

1. *Develop*: AI generates code, human defines specs and intent
2. *Review*: Quality gates (gmake check), code review, progressive commits
3. *Harden*: Security fixes, contract audits, documentation updates

** Where comprehension debt enters each phase

| Phase | Debt risk | Mitigation present | Gap |
|-------|-----------|-------------------|-----|
| Develop | AI generates faster than understanding | Spec in CLAUDE.md | No comprehension gate |
| Review | Tests pass without understanding | gmake check, 42+ tests | No "explain why" step |
| Harden | Fixes applied without root-cause trace | Security tests, ADRs | Only 5 ADRs for 21 modules |

** Proposed additions

1. *Develop phase*: Add an "intent document" requirement for any module
   larger than 200 lines.  Record: what problem does this module solve,
   what are its invariants, what should it /never/ do.

2. *Review phase*: Add a "comprehension gate" to the eval harness.
   After generating a code change, the model must answer 3 comprehension
   questions about the affected modules.  Human reviewer scores the
   answers.

3. *Harden phase*: Monthly "theory rebuild" --- walk through one major
   module per session, reconstructing the data flow and design rationale.
   Document findings in ADRs.

* Toward a Comprehension Score

** Why no metric exists yet

The literature converges on a single conclusion: there is no standardized
comprehension metric analogous to code coverage.  Current velocity
metrics (DORA, PR counts, lines changed) are not just insufficient ---
they are /actively misleading/ because they reward the behaviors that
generate comprehension debt.

** Candidate proxy metrics

Drawing from the literature survey, a "comprehension score" for a
codebase might combine:

| Component | Weight | How to measure |
|-----------|--------|---------------|
| ADR coverage | 15% | ADRs per module (target: 1 per major module) |
| Comment density | 10% | Weighted by module complexity (lines x fan-in) |
| Test-with-intent ratio | 20% | % of tests that document "why", not just "what" |
| Untested module ratio | 15% | LOC without dedicated tests / total LOC |
| Comprehension quiz pass rate | 25% | Eval harness "why" questions (Section 5) |
| Incident MTTR trend | 15% | Time-to-root-cause for production issues |

** guile-sage self-assessment (current state)

| Component | Current | Target | Status |
|-----------|---------|--------|--------|
| ADR coverage | 5 ADRs / 21 modules (24%) | 1 per major module (60%) | Below |
| Comment density | 9-35%, median ~20% | 20% minimum everywhere | Partial |
| Test-with-intent ratio | Low (tests check behavior, rarely document why) | 50%+ | Below |
| Untested module ratio | 40% of LOC untested | <15% | Below |
| Comprehension quiz pass rate | Not measured | 80%+ | N/A |
| Incident MTTR | Not measured (no production) | Baseline | N/A |

Estimated comprehension score: *35-40%* (well below a healthy threshold
of 70%+).

* Key Findings and Recommendations

** Finding 1: The comprehension gap is real and measurable

The Anthropic study (17% comprehension drop), the epistemic debt study
(69.3 points debt in unrestricted AI conditions), and the 5-7x
velocity-comprehension gap all point to the same phenomenon:
AI-generated code accumulates faster than human understanding.  This
is Brooks' essential/accidental distinction ("No Silver Bullet", 1986)
playing out with new tooling: AI removes accidental complexity
(typing) without touching the essential complexity of understanding
what the code means.

** Finding 2: guile-sage has significant comprehension debt

With 83% AI co-authored commits, 40% untested module LOC, the largest
module (repl.scm) at 9% comment density, and only 5 ADRs for 21
modules, guile-sage exhibits classic comprehension debt indicators.
The test suite is strong in coverage breadth (24 suites) but weak in
comprehension depth (no "why" testing).

** Finding 3: The eval harness needs a comprehension dimension

The current eval harness tests "correct tool call" (traffic cases) and
"correct code generation" (algorithmic problems).  It does not test
"correct understanding."  Adding rationale probes, counterfactual
questions, and dependency tracing would create a more complete picture.

** Finding 4: The Explanation Gate pattern works

The epistemic debt study showed that requiring articulation before
integration reduced debt by 60% with only 3% loss in utility.  This is
directly applicable to the sage eval harness and to the agentic
development workflow more broadly.

** Finding 5: Observability is the runtime complement of comprehension

Charity Majors' insight applies: you cannot understand a system by
reading code alone.  guile-sage already emits telemetry counters
(session.count, token.usage, etc.) to the OTLP stack.  Extending this to
include /comprehension-relevant/ signals (e.g., time spent in review,
number of "why" questions asked during development) would close the loop.

** Recommendations

1. *Immediate*: Add 5-10 comprehension eval cases to traffic-cases.json
   (rationale probes, counterfactual questions, dependency tracing)

2. *Short-term*: Write ADRs for the 6 largest modules without them
   (repl, tools, mcp, ollama, util, logging)

3. *Short-term*: Add test files for gemini.scm and openai.scm (the two
   untested provider modules)

4. *Medium-term*: Implement an Explanation Gate in the eval harness:
   after tool selection, require the model to explain /why/ before
   scoring.

5. *Medium-term*: Establish a monthly "theory rebuild" session, walking
   through one major module to reconstruct and document its design
   rationale.

6. *Long-term*: Define and track a composite comprehension score,
   reporting it alongside test coverage in CI output.

7. *Cultural*: Budget 20-30% additional time for comprehension in
   AI-assisted development sessions, as recommended by Pellon.
