* 4. Findings & roadmap inferences (best-effort, native Claude Code + Codex)

*Headline: guile-sage should BE an MCP server, not just a client.* Both Claude
Code and Codex are MCP clients; *Codex already runs as an MCP server*
(=codex mcp-server=). The point of this review isn't to match their agent-UX
feature-for-feature — it's to expose sage's own surface (tool registry, sandbox,
eval, taint boundary) over MCP so those agents *call sage*. That turns
guile-sage into the governed tool/sandbox *substrate* other agents run on, reusing
the L3 assets we already own (=*tools*=/=register-tool= in =tools.scm=; JSON-RPC +
transport in =mcp.scm=). Gap = the server-side =tools/list=/=tools/call= loop.
Prioritized: =SAGE-t2l=; see [[file:../ROADMAP.org][ROADMAP.org]] "be the MCP server".

What else the two natively-runnable agents tell us for guile-sage's H-layer:

1. *Sandbox + permission model is table-stakes.* Codex ships a first-class
   =codex sandbox= subcommand with =sandbox_permissions= (disk-access levels);
   Claude Code has permission-modes + "recommended only for sandboxes with no
   internet" guidance. Both validate RFC-003's direction. *Concrete to mirror:*
   Codex's tiered =sandbox_permissions= (e.g. disk-full-read-access) is a clean
   model for our HC-01 fs-confinement levels.
2. *MCP is universal — and "be an MCP server" is emerging.* Both are MCP
   clients; *Codex can also run AS an MCP server* (=codex mcp-server= stdio).
   guile-sage is client-only; an MCP-server mode (we have it only as
   forward-looking LSP) would let other agents call sage's tools. Roadmap candidate.
3. *First-class sub-agents.* Claude Code takes =--agents= JSON to define custom
   sub-agents inline; Codex has session =fork=. guile-sage's multiagent is
   =P= (aq/coordinator) — promoting sub-agents to a first-class, inline-definable
   surface is a gap worth a bead.
4. *Session fork/resume.* Codex =resume=/=fork=, Claude =--continue=. guile-sage
   has session persistence (=Y=) but no explicit *fork* (branch a session) —
   small, high-value add.
5. *Built-in code review verb.* =codex review= (non-interactive) is a discrete
   capability; pairs with our eval lineage. Consider a =sage review= path.
6. *Cloud/remote tasks.* Codex =cloud= + =--remote= app-server; out of scope for
   a self-hosted/LAN posture, but notable as a direction the field is taking.
7. *Platform reality:* Claude Code + Codex run on FreeBSD; OpenCode + Droid do
   not — FreeBSD support is a guile-sage differentiator. Finish the matrix's
   OpenCode/Droid columns from a Linux/macOS host (=SAGE-4fl=).

These are =--help=-level inferences, not a hands-on feature test — good enough to
steer the roadmap; deepen per-feature if a row drives a real decision.
