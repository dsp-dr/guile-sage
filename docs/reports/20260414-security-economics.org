* Executive Summary

AI-driven cybersecurity has become an economic arms race. Drew Breunig's
"proof-of-work" framing (April 2026) reduces security to a brutally
simple equation: /defenders must spend more tokens discovering
vulnerabilities than attackers spend exploiting them/. AISI's evaluation
of Claude Mythos Preview --- 100M tokens per attack attempt, $12,500
per run, 3/10 success rate on a 32-step corporate network breach ---
provides the first empirical cost data for this model.

This report surveys the evidence, maps it to sage's threat surface,
and proposes a concrete hardening budget for the v0.7.x and v1.0
release cycles.

* The Proof-of-Work Thesis

** Breunig's Core Argument

Drew Breunig ("Cybersecurity Looks Like Proof of Work Now", April 14
2026) argues that AI-powered vulnerability discovery has transformed
security from a cleverness game into a spending game:

#+BEGIN_QUOTE
Buy the tokens, maybe you find an exploit. Hopefully you keep trying
longer than your attackers.
#+END_QUOTE

The central formula: *defending systems requires spending more tokens
discovering vulnerabilities than adversaries will spend exploiting
them*. This mirrors cryptocurrency's proof-of-work, where success
correlates with raw computational expenditure rather than strategic
insight.

** The Three-Phase Cycle

Breunig predicts hardening emerges as a distinct third phase in
agentic software development:

| Phase       | Limiter        | Activity                                    |
|-------------+----------------+---------------------------------------------|
| Development | Human input    | Feature iteration, guided by intuition       |
| Review      | Async capacity | Refactoring, documentation, standards        |
| Hardening   | Token budget   | Autonomous exploit discovery until funds run out |

The first phase is limited by human attention. The third is limited
purely by money. This inversion --- security as a budget line item
denominated in tokens rather than headcount --- changes how teams
plan releases.

** Open Source Economics

Willison amplifies Breunig's observation that open-source libraries
gain unexpected strategic value under this model: tokens spent
securing widely-used packages are amortized across all their users.
Conversely, popular OSS packages become higher-value attack targets
precisely because of their reach.

* Empirical Evidence: What Attacks Actually Cost

** AISI's "The Last Ones" Benchmark

The UK AI Safety Institute built "The Last Ones" (TLO): a 32-step
corporate network attack simulation spanning reconnaissance through
full network takeover, estimated to require human professionals 20
hours to complete.

*** Token Budget vs Performance (TLO step completion)

| Model              | 10M tokens (avg steps) | 100M tokens (avg steps) |
|--------------------+------------------------+-------------------------|
| GPT-4o (Aug 2024)  |                    1.7 | ---                     |
| Sonnet 3.7 (Feb 25)|                    5.8 | ---                     |
| Sonnet 4.5 (Sep 25)|                    6.1 |                     9.4 |
| 5.1 Codex (Nov 25) |                    8.0 |                     9.8 |
| Opus 4.5 (Nov 25)  |                    7.6 |                    11.0 |
| Opus 4.6 (Feb 26)  |                    9.8 |                    15.6 |
| Mythos Preview      | ---                    |                    22.0 |

Mythos Preview completed all 32 steps in 3 of 10 attempts.

*** Scaling Properties

Performance scales *log-linearly* with inference-time compute:

- 10M to 100M tokens yields gains of *up to 59%*
- *No observed plateau* at any tested budget
- Models were still executing goal-directed commands when token
  limits were reached
- A 100M-token attempt with Opus 4.6 costs approximately *$80 USD*
  (with input caching)
- Mythos at 100M tokens costs *$12,500 per attempt*

*** Critical Limitation

The TLO benchmark /lacks active defenders/, security monitoring, and
penalties for triggering alerts. Real-world systems with layered
defenses would impose additional costs on attackers --- but the
log-linear scaling relationship means attackers can brute-force
through defenses by spending more tokens.

* The Supply Chain Dimension

** LiteLLM: A Direct Wake-Up Call

On March 24 2026, compromised LiteLLM packages (v1.82.7, v1.82.8)
were published to PyPI and were live for 40 minutes. The attack
chain demonstrates Breunig's model in reverse --- attackers invested
tokens in a multi-hop supply chain attack:

1. *Trivy compromise* (March 19): Force-pushed malicious code to 75
   of 76 version tags in =aquasecurity/trivy-action=
2. *CI/CD pivot*: Compromised Trivy exfiltrated =PYPI_PUBLISH= token
   from LiteLLM's GitHub Actions runner
3. *Package poisoning*: Malicious =.pth= file in litellm executed on
   every Python process startup
4. *Payload*: Exfiltrated SSH keys, AWS/GCP/Azure creds, Kubernetes
   configs, git credentials, shell history, crypto wallets, SSL
   private keys, CI/CD secrets, database passwords

/guile-sage uses LiteLLM as its proxy gateway/. We were not affected
(we pin to v1.83.4 and did not install during the 40-minute window),
but this is our #1 supply chain risk.

** Axios: State-Actor Supply Chain Attack

On March 31 2026, the popular HTTP client Axios (npm) was
compromised via a North Korean state actor (Sapphire Sleet /
UNC1069). Backdoored versions (1.14.1, 0.30.4) delivered a
cross-platform RAT. Malicious versions were live for ~3 hours.

** Karpathy's "Yoink" Argument

In response to the LiteLLM attack, Andrej Karpathy argued that
modern dependency pyramids must be re-evaluated:

#+BEGIN_QUOTE
Classical software engineering would have you believe that
dependencies are good (building pyramids from bricks), but this
has to be re-evaluated.
#+END_QUOTE

His position: use LLMs to "yoink" (reimplement) needed
functionality rather than importing entire dependency trees. This
reduces the attack surface but increases the maintenance surface.

*Relevance to sage*: guile-sage has minimal dependencies (GNU Guile
standard library plus Guile JSON). This is a structural advantage.
The primary supply chain risk is the LiteLLM proxy itself, which
runs as a separate process.

* Trail of Bits: The AI-Native Audit Model

Trail of Bits provides the closest real-world analog to what sage's
hardening phase could look like:

| Metric                    | Before AI    | After AI (2026)    |
|---------------------------+--------------+--------------------|
| Bugs found per week       | ~15          | ~200               |
| AI-originated bugs        | 0%           | 20% of all reports |
| Tooling infrastructure    | Manual       | 94 plugins, 201 skills, 84 agents |
| Revenue per sales rep     | $2--4M (industry avg) | $8M        |

Their approach: specialized agents run full analysis passes in
parallel, producing findings/triage/report drafts. Human auditors
validate before client delivery. The key insight: /AI-augmented
continuous scanning replaces periodic discrete audits/.

They note that traditional audit scoping, pricing, and deliverables
are all "on the table" for restructuring because AI-augmented
auditors create order-of-magnitude productivity differences.

* Cost Curves and Diminishing Returns

** The Log-Linear Question

AISI's data is unambiguous for /offensive/ token spending: no
diminishing returns were observed up to 100M tokens. But offensive
and defensive scanning may have different curves:

- *Offensive* (finding any one exploit): log-linear, because the
  attacker needs only one success path through a large space
- *Defensive* (finding all exploits): likely steeper diminishing
  returns, because each successive vulnerability is harder to find
  in a shrinking residual space

** Smart Contract Evidence

Trail of Bits' smart contract data shows diminishing returns in
iterative exploit generation:

| Iteration | Marginal gain |
|-----------+---------------|
|         2 | +9.7%         |
|         3 | +3.7%         |
|         4 | +5.1%         |
|         5 | +2.8%         |

Most successful exploits emerged within 5 iterations, suggesting a
practical saturation point for structured vulnerability classes.

** The Asymmetry Problem

Economic analysis reveals a troubling asymmetry:

- Attackers achieve profitability at *$6,000* exploit values
- Defenders require *$60,000* to match

This 10x asymmetry means defenders must either:
1. Accept partial coverage and prioritize high-value targets
2. Amortize costs across shared infrastructure (open source model)
3. Use cheaper local inference for breadth, API for depth

|-----------------------+------------------+-------------------+---------------|
| Opus 4.6 (API)        | $5.00            | $25.00            | ~$80 (cached) |

Local inference is *6,000x cheaper per token* than API pricing.
For breadth-first scanning, local models dominate. For depth

are necessary.

** What an Attacker Would Spend

At Opus 4.6 pricing ($80/100M tokens), an attacker targeting sage
specifically would need:

- *Reconnaissance* (reading source, understanding architecture): ~1M
  tokens, ~$0.80
- *Exploit generation* (finding sandbox escapes, injection vectors):
  ~10M tokens, ~$8
- *Validation* (confirming exploits work end-to-end): ~5M tokens, ~$4

Total: roughly *$15--$50 per attack campaign* against sage's codebase.

With local Ollama inference, this drops to near-zero (minutes of GPU
time on consumer hardware). The implication: /any motivated attacker
can afford to probe sage's security surface exhaustively/.

* Recommendations: Hardening Budget for Sage

** Principle: Outspend the Attacker

If an attacker can mount a thorough campaign for $50 in API tokens
(or near-zero with local inference), sage's hardening budget must
exceed this by a meaningful margin. The goal is not parity but
/deterrence through depth/ --- ensuring that obvious vulnerabilities
are found before attackers look.

** Continuous vs Discrete: The Hybrid Model

The evidence supports a hybrid approach:

*** Continuous (every commit)
- Low-cost, high-frequency scanning via local Ollama inference
- Catches regressions immediately
- ~2--3M tokens per commit (effectively free with local inference)
- Integrated into =gmake check= or pre-push hooks

*** Discrete (per release)
- Deeper API-powered scans at 10--50M token budgets
- Catches multi-step exploits and subtle logic errors
- ~35M tokens per minor release, ~150M for major releases
- Run as a release gate, results archived in =docs/reports/=

*** Annual External
- One traditional security audit per year ($10--50K)
- Validates that automated scanning has not developed blind spots
- Required for any compliance certification

* Key Questions Answered

** 1. What's the right hardening budget for sage?

*35M tokens per minor release, 150M for v1.0.* At current Opus 4.6
pricing, this is $28--$120 per release. With local inference, it is
effectively free. The budget should exceed the estimated attacker
cost ($15--$50) by at least 3x.

** 2. Is continuous hardening (every commit) better than discrete audits?

*Both, in a hybrid model.* Continuous low-cost scanning (2--3M
tokens/commit via local inference) catches regressions. Discrete
deep scans (35--150M tokens) at release boundaries catch multi-step
exploits. Neither alone is sufficient.

** 3. How do we operationalize "spend more tokens finding exploits than attackers spend exploiting them"?

Three mechanisms:
1. *Instrument*: track =security.scan.tokens= in telemetry
2. *Gate*: block releases below minimum token thresholds
3. *Amortize*: open-source the security test cases so the
   community's token spending adds to our defense budget

** 4. What's the cost curve --- do we hit diminishing returns at 10M? 100M?

AISI's offensive data shows *no plateau up to 100M tokens*. However,
for /defensive/ scanning of a small codebase like sage (~8K lines),
diminishing returns likely set in earlier:
- *1--10M tokens*: catches commodity vulnerabilities (path traversal,
  injection, permission bypass)
- *10--50M tokens*: catches logic errors, multi-step chains
- *50--150M tokens*: catches subtle architectural issues
- *>150M tokens*: diminishing returns for sage's current size

The smart contract evidence suggests most exploits emerge within
5 iterations, with marginal gains dropping from +9.7% to +2.8%.

** 5. Should the eval harness include adversarial/red-team test cases?

*Yes, absolutely.* This is the single highest-ROI investment. Adding
10 adversarial cases (SEC01--SEC10) to the existing eval harness
converts the hardening phase from ad-hoc to systematic. Each case
encodes a specific threat from the OWASP LLM Top 10 and runs on
every =gmake check=.
