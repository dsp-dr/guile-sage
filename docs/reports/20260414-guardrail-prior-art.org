* Preamble

I read the companion report ("Guardrail Architectures for Tool-Calling LLM
Agents", 2026-04-14) with the mixture of admiration and exasperation that I
reserve for well-executed work that has absolutely no sense of where it came
from.  Eight systems surveyed.  Token overhead priced out.  Scheme code
sketched.  Thorough, current, and almost entirely ahistorical.

Not a single citation predates 2023.  Not one.

The report describes proxy-based interposition, mandatory content filtering,
capability-restricted tool invocation, layered trust hierarchies, and
validate-before-execute patterns---and treats each as if it sprang fully
formed from the forehead of a Y Combinator startup.

They did not.  This counter-report identifies the prior art.  It is opinionated.

* 1. Reference Monitors (Anderson, 1972)

Anderson's 1972 "Computer Security Technology Planning Study" defined the
/reference monitor/: a tamper-proof, always-invoked, verifiable mediation
layer through which all access requests must pass.  Three properties:

1. *Complete mediation*: every access is checked, no bypass path exists
2. *Tamper-proof*: the monitor cannot be modified by its subjects
3. *Verifiable*: small enough for exhaustive analysis

LiteLLM's proxy is a reference monitor.  NeMo's rails are a reference
monitor.  Azure's gateway policy is a reference monitor.  /None/ satisfy
Anderson's three properties:

- *Complete mediation*: the companion report itself documents the gap---tool
  execution results bypass the proxy.  Anderson would have called this a
  TOCTOU violation in 1972.
- *Tamper-proof*: an LLM can be jailbroken into skipping its own guardrail
  calls.  The monitor is only tamper-proof if the subject cannot influence
  whether it is invoked.
- *Verifiable*: a Python callback with 500 lines of async logic is not
  verifiable.  Anderson envisioned formal proof.  We got =async_pre_call_hook=.

Fifty-four years of insight, discarded because it was published before anyone
in the room was born.

* 2. Capability-Based Security (Dennis & Van Horn, 1966)

Dennis and Van Horn's 1966 paper introduced /capabilities/: unforgeable tokens
conferring specific access rights.  A capability is not a permission /check/---it
is the permission /itself/.

LiteLLM's "Tool Permission Guardrail" controls tool invocation via allow/deny
rules.  This is an /access control list/ bolted onto a proxy.  The distinction
matters:

- ACLs answer "who can access this resource?" (ambient authority)
- Capabilities answer "what can this subject do?" (no ambient authority)

Hardy's confused deputy problem (1988) showed ACL systems are vulnerable to
agents being tricked into using their ambient authority for an attacker.  This is
/exactly/ indirect prompt injection.  Dennis and Van Horn's capabilities---and
Miller's object-capability refinement (2006)---solve it by construction.

sage's =*safe-tools*= list is an ACL.  A capability-based design would pass
unforgeable tool tokens into the execution context, making it /structurally
impossible/ for an injected prompt to invoke an unauthorized tool.  That is not
an academic distinction.  It is the difference between secure-by-policy and
secure-by-construction.

* 3. Bell-LaPadula and Biba: Information Flow (1973, 1977)

Bell and LaPadula formalized mandatory access control in 1973:

1. *No read up*: a subject cannot read above its clearance
2. *No write down*: a subject cannot write below its clearance

The companion report's layered architecture is a lattice-based information flow
model wearing a trench coat:

#+begin_example
Layer 4: Model-Intrinsic Safety     [TOP SECRET]
Layer 3: Agent-Side Output Validation [SECRET]
Layer 2: Agent-Side Input Validation  [CONFIDENTIAL]
Layer 1: LiteLLM Proxy               [UNCLASSIFIED]
#+end_example

Tool outputs must not flow into the LLM context without a declassification
gate.  That is Bell-LaPadula's star property.  The PII regex screen is a
/downgrader/.  Bell-LaPadula gives a rigorous framework for reasoning about
when downgrading is safe.  Regex gives us hope and prayer.

Biba (1977) provided the integrity dual.  "No write up, no read down" maps
directly onto indirect prompt injection: untrusted tool outputs (low integrity)
must not influence the LLM's behavior (high integrity context).  None of the
eight surveyed systems formalize this.  They should.

* 4. Saltzer & Schroeder's Eight Principles (1975)

"The Protection of Information in Computer Systems" gave us eight design
principles.  Least privilege is only one.  Consider how many others the
guardrail industry violates:

- *Economy of mechanism*: NeMo adds a Colang DSL, a flow engine,
  chain-of-thought prompting, and a microservice.  Saltzer would weep.
- *Fail-safe defaults*: Ollama/local models have "no intrinsic safety."
  Defaulting to open when guardrails are absent is not fail-safe.
- *Open design*: Constitutional AI depends on proprietary training data
  and opaque classifier weights.  Security through obscurity in a lab coat.
- *Psychological acceptability*: Gemini "sometimes over-refuses."  A guardrail
  that blocks legitimate use is a guardrail users will route around.  Saltzer
  and Schroeder predicted this in 1975.

* 5. Proof-Carrying Code (Necula, 1997)

Necula proposed that untrusted code carry a machine-checkable proof of its
safety properties.  The consumer verifies the proof (cheap) rather than
re-analyzing the code (expensive) or trusting the producer (dangerous).

Every guardrail system in the companion report re-analyzes every response.
Every time.  This is the expensive path PCC was designed to avoid.

Structured output schemas (JSON mode, function calling) are primitive PCC:
the LLM produces output carrying evidence of structural validity, and the
consumer verifies the schema rather than parsing arbitrary text.  The gap
is that nobody extends this to /semantic/ safety.  We verify JSON schemas but
not "this response contains no social scoring recommendations."  Necula showed
us how thirty years ago.  We chose regex.

* 6. TCSEC: The Orange Book (DoD, 1983)

The Trusted Computer System Evaluation Criteria defined assurance levels from D
(minimal) through A1 (verified design).  Security is a spectrum; higher
assurance requires formal methods, not just more testing.

Map the companion report's systems to TCSEC levels:

| System | Approximate TCSEC Level |
|--------+-------------------------|
| Regex/keyword filters | C1 (discretionary) |
| LLM-based moderation | C2 (controlled access) |
| NeMo Colang rails | B1 (labeled security) |
| Constitutional AI | B2 (structured protection) |
| Formal verification | A1 (verified design) |

Nobody is at A1.  Nobody is trying.  The industry ships C1 assurance at B2
prices for systems handling PII, credentials, and EU-regulated content.

* 7. Wittgenstein's Language Games (1953)

Wittgenstein argued that meaning is use within a /language game/---context-
dependent, rule-governed.  No fixed universal meaning; meaning is constituted
by practice.

The companion report documents the failure: keyword guardrails use "social
scoring" but users say "citizen trustworthiness scoring system."  The guardrail
designers specified /words/ when they needed to specify /language games/.

A keyword list is a theory of meaning based on /reference/.  But any
/extensional/ definition (a list of prohibited things) is always incomplete.
What is needed is an /intensional/ definition: why something is prohibited,
from which instances can be derived.  Anthropic's Constitutional AI gestures
at this with "reason-based alignment," but buries the reasoning in opaque
weights rather than making it inspectable.

* 8. Popper's Falsificationism (1934/1959)

Popper argued that theories cannot be verified but can be falsified.  Good
theories are maximally falsifiable.

Guardrail systems are Popperian falsification machines.  They attempt to
/disprove/ safety (find a PII match, a toxicity hit, a keyword violation).
If no counterexample is found, the response is provisionally accepted---not
proven safe.  But the industry does not acknowledge this asymmetry:

- Passing PII filters means "PII-free w.r.t. patterns we checked"
- Passing toxicity means "non-toxic per this classifier's training data"
- Passing red-team testing means "secure against the attacks we tried"

The companion report's claim that regex + keyword filters "cover the 90%
case" is unfalsifiable as stated---90% of /what/?

* 9. Lampson's Confinement Problem (1973)

Lampson showed that preventing information leakage is fundamentally hard.
Covert channels---timing, storage, resource consumption---allow information
to escape even when explicit channels are controlled.

LLM agents have covert channels the guardrail industry ignores entirely:

- *Timing*: response latency varies with content
- *Token count*: response length encodes information even when content is clean
- *Tool selection*: which tool is invoked leaks prompt information
- *Refusal patterns*: what the model refuses reveals its training

Lampson proved complete confinement is impossible in the general case.  The
industry's implicit promise of containment is provably unachievable.

* Conclusion: What a Historically Informed Guardrail Would Look Like

If the industry read its own history, it would build:

1. *Anderson's reference monitor*: complete mediation (no tool output bypass),
   tamper-proof, verifiable (small enough to formally analyze)
2. *Dennis & Van Horn's capabilities*: unforgeable tool tokens, not ACLs;
   confused deputy eliminated by construction
3. *Bell-LaPadula/Biba flow control*: formal lattice reasoning about when
   untrusted data may enter trusted contexts
4. *Necula's PCC*: responses carry safety certificates; verification is O(n)
5. *Saltzer & Schroeder*: economy of mechanism, fail-safe defaults, open design
6. *Popper's falsificationism*: state corroboration degree, not safety level
7. *Wittgenstein*: define prohibitions intensionally, not extensionally

None of the eight surveyed systems achieve more than two of these.  Most
achieve zero.

The LLM guardrail industry is fifty years behind computer security, thirty
years behind formal methods, and seventy years behind philosophy of language.
It has venture capital.  It does not have intellectual foundations.

Build the foundations.  Read the papers.  Or at minimum, stop acting surprised
when you rediscover what Lampson and Bell and Saltzer already knew.
