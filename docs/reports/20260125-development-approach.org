* Introduction

This document describes the development approach used to build guile-sage,
an AI REPL with tool calling capabilities in Guile3 Scheme.

* Inspiration and Source Analysis

** sage.el (aygp-dr/sage.el)

The primary inspiration was sage.el, an Emacs package providing multi-provider
LLM integration. Key features analyzed:

- Multi-provider architecture (Ollama, OpenAI, Anthropic, etc.)
- Tool calling with safety permissions
- Session management with persistence
- Context window management
- Streaming responses

We adopted:
- Tool calling patterns
- Session structure (messages, stats, timestamps)
- Safety permission model (safe vs unsafe tools)

** gemini-repl-009 (aygp-dr/gemini-repl-009)

Requirements documentation that informed our feature set:

- Phase-based development (Core, Tools, Context, Session)
- Tool schema definitions
- Context compaction strategies
- Testing requirements

** Claude Code System Prompts (Piebald-AI)

Studied Claude Code's approach to:
- System prompt structure and best practices
- Tool descriptions and usage guidelines
- Path safety and workspace isolation
- Project directory encoding (-home-user-project style)

* Development Methodology

** Beads Methodology

We followed the "beads" methodology for development:

1. *Small Commits* - Each logical change is its own commit
2. *Atomic Changes* - Complete and reversible
3. *Progressive Protocol* - Build feature by feature
4. *Git Notes* - Document issues and decisions

** Platform-Specific Workarounds

FreeBSD 14.3 presented unique challenges:

1. *popen Segfault* - Guile3's =open-input-pipe= caused segfaults
   - Solution: Use =system()= with temp files for curl operations

2. *Missing guile-json* - No JSON library available
   - Solution: Implement custom JSON parser/writer

** Tool Permission Model

#+begin_src scheme
;; Safe tools - always allowed
(define *safe-tools*
  '("read_file" "list_files" "git_status" "git_diff" ...))

;; Permission check
(define (check-permission tool-name args)
  (or (member tool-name *safe-tools*)
      (config-get "YOLO_MODE")
      #f))
#+end_src

Rationale:
- Default-deny for unsafe operations
- YOLO mode for trusted environments
- Clear boundary between read/write operations

** XDG Compliance

#+begin_src scheme
(define (xdg-data-home)
  (or (getenv "XDG_DATA_HOME")
      (string-append (getenv "HOME") "/.local/share")))

(define (sage-data-dir)
  (string-append (xdg-data-home) "/sage"))
#+end_src

Rationale:
- Standard directory structure
- Override via environment variables
- Clean separation of config/data/cache/state

** Project Path Encoding

#+begin_src scheme
;; /home/user/project -> -home-user-project
(define (path->project-id path)
  (string-map (lambda (c) (if (char=? c #\/) #\- c)) path))
#+end_src

Rationale:
- Matches Claude Code behavior
- Per-project session isolation
- Valid filesystem names

* Tool Implementation Patterns

** Tool Registration

#+begin_src scheme
(register-tool
 "read_file"
 "Read contents of a file within workspace"
 '(("type" . "object")
   ("properties" . (("path" . (("type" . "string")
                               ("description" . "File path")))))
   ("required" . #("path")))
 (lambda (args)
   (let ((path (assoc-ref args "path")))
     (if (safe-path? path)
         (call-with-input-file (full-path path) get-string-all)
         (format #f "Unsafe path: ~a" path)))))
#+end_src

** Self-Modification Tools

Critical for AI self-improvement:

1. *eval_scheme* - Evaluate arbitrary Scheme code
2. *create_tool* - Register new tools dynamically
3. *reload_module* - Hot-reload modified modules
4. *edit_file* - Modify source files

* Testing Strategy

** Unit Tests

Each module has comprehensive unit tests:

#+begin_src scheme
(run-test "read_file reads fixture"
  (lambda ()
    (let ((result (execute-tool "read_file"
                               '(("path" . "fixtures/test.txt")))))
      (unless (string-contains result "test file")
        (error "content not found" result)))))
#+end_src

** Test Modes

Tests adapt to YOLO mode:

#+begin_src scheme
(run-test "write_file permission depends on mode"
  (lambda ()
    (let ((allowed (check-permission "write_file" '())))
      (if *yolo-mode*
          (unless allowed (error "should be allowed in YOLO"))
          (when allowed (error "should be blocked"))))))
#+end_src

* Lessons Learned

1. *Platform Testing* - FreeBSD revealed issues not seen on Linux
2. *JSON Independence* - Don't assume library availability
3. *Safety First* - Default-deny is better than default-allow
4. *Incremental Testing* - Test each feature as it's built
5. *Documentation* - Org-mode keeps docs close to code

* Future Considerations

1. *Streaming Responses* - For long generations
2. *Multi-Provider* - Beyond Ollama
3. *MCP Protocol* - Standard tool interop
4. *Agent Teams* - Multi-agent coordination
5. *IDE Integration* - Beyond CLI
