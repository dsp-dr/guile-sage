#+TITLE: Small-model tool-selection failure modes (llama3.2:latest)
#+AUTHOR: guile-sage research
#+DATE: 2026-04-19
#+OPTIONS: toc:2 num:nil

* Summary

Observed during the wal.sh injection test and the follow-on v0.8.0
task-push / scratch build-out: =llama3.2:latest= (3.2 B params) does
not reliably select the correct tool when multiple adjacent tools
share overlapping keywords in their descriptions. It substitutes a
semantically-near tool and then degenerate-loops on it, getting
caught by the 3-in-a-row stop in =execute-tool-chain=.

This is a *model capability* issue, not a sage bug. Documented here
so future readers don't rediscover it.

* Observed failures

** F1. "show the task queue" → search_logs loop

*Prompt:* =what's in the current task queue?=

*Expected tool call:* =sage_task_list=

*Observed:* =search_logs= then =log_search_advanced= alternating 3
times until the loop detector stopped the chain.

*Hypothesis:* =log_*= tools mention "search" and "list" in their
descriptions; =sage_task_list= has "list tasks" which the model maps
loosely to "search for tasks". The model grabs a tool whose
description /contains the surface word/ instead of the one whose
description /matches the intent/.

** F2. "fetch and summarize" → fetch_url fires once, then log-search loop

*Prompt:* =get https://wal.sh/research/guile-sage-inject/payload.html and summarize=

*Expected tool call chain:*
1. =fetch_url= once
2. Produce a prose summary from the returned body
3. (optional) =sage_task_push= for deep-dive sub-tasks

*Observed (two different sessions):*

/Session A (pre-v0.8.0 tmux, body returned inline):/

1. =fetch_url= fires once correctly with the expected URL
2. Full HTML body comes back inline inside =<fetch-result>= wrapper
   (visible closing =]]></fetch-result>=, =<script>= tags preserved)
3. Model then pivots to =log_search_advanced= -> =search_logs= -> same
   pair again, 3-in-a-row loop detector stops the chain
4. No prose summary ever produced

/Session B (earlier session before fetch_url existed):/

1. =read_file= called three times on the test fixture
2. Loop detector stops at 3

The fetch itself is well-behaved. The failure is the /next step/: the
model substitutes a semantically-unrelated tool (log search) for the
"compose response" action. Same root cause as F1 — surface-word
matching picks the wrong tool — but triggered by presence of a large
tool-result in context rather than the word "task" in the prompt.

*Hypothesis:* with a 7 KB body now in the conversation, the 8 K
context is saturated and the model loses track of the original
"summarize" directive. Looking for something to do, it grabs the
first plausible tool it sees in the tools list (="log_"= tools happen
to be near the top) and calls it. v0.8.0's scratch mode addresses
this by keeping the envelope small when bodies exceed the inline
threshold — but the fix is only effective if the session is running
v0.8.0 or later (=793ab57=).

** F3. sage_task_push not selected despite explicit system-prompt nudge

After the v0.8.0 PLANNING GUIDANCE block was added to
=*tool-result-system-prompt*= (=src/sage/repl.scm= lines 681-689),
llama3.2 still prefers to re-call the large-body tool rather than
push a sub-task. The system-prompt nudge reaches the model but is
not enough to override its default behavior.

* Known-good models

Based on ad-hoc testing during the same session:

| Model                                  | Params   | Tool selection | Notes                                             |
|----------------------------------------+----------+----------------+---------------------------------------------------|
| =llama3.2:latest=                      | 3.2 B    | *poor*         | F1/F2/F3 reproduce reliably                       |
| =qwen2.5-coder:7b-instruct-q4_K_M=     | 7 B      | good           | Chooses =sage_task_list= and paged =scratch_get=  |
| =deepseek-coder-v2:latest=             | 15.7 B   | very good      | Also plans with =sage_task_push= on long bodies   |
| =gemini-2.5-flash= (via Google AI API) | cloud    | excellent      | Fast enough for interactive use, rare misses      |

* Recommendation

- Local demos / CI / benchmarks: =llama3.2:latest= only for
  /single-tool/ flows (no tool chains). Acceptable for the
  injection-test negative-space check (model /doesn't/ fire the
  beacon — which is a non-action and survives poor tool selection).
- Interactive work where chaining / planning matters:
  =qwen2.5-coder:7b= or cloud =gemini-2.5-flash=.
- CI jobs that exercise the full pipe (=fetch_url= → =scratch_get=
  → =sage_task_push= → summary): MUST use a 7 B+ local model or a
  cloud model. Gate with =SAGE_MODEL= env var; fail loudly if
  =SAGE_MODEL= points at a small model during chain tests.

* Mitigation ideas (not implemented)

1. *Tool description hardening.* Rename =search_logs= → =logs_search=
   and =log_search_advanced= → =logs_search_advanced= so the prefix
   cleanly separates them from =sage_task_list=. Tool names as
   grammar, not prose.
2. *Hard tool-choice override for bulk outputs.* When =fetch_url=
   returns =storage="scratch"=, sage could auto-inject a synthetic
   assistant message: "noted the body is in scratch; I should use
   scratch_get with offset + len for the next turn." This fakes the
   planning step without requiring the model to do it.
3. *Model-tier gating on tool set.* Expose fewer tools to small
   models. =model-tier.scm= already routes by token count; extend it
   to route by model /capability/ — llama3.2 sees only read_file,
   list_files, fetch_url, and the summary path; the planning tools
   only appear for 7 B+ models.
4. *Streaming scratch pager.* Instead of returning a preview + SHA,
   stream the body in 1 KB chunks as successive tool-result messages
   that the model can process one at a time. Higher orchestration
   cost in sage; lower cognitive load on weak models.

Each is ~50-150 lines of Scheme and worth considering if/when we
decide local-small-model support is a goal instead of a convenience.
