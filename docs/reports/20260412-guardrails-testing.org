End-to-end testing of LiteLLM guardrails through guile-sage 0.7.0.
Sage talks to Gemini 2.5 Flash via the LiteLLM proxy at
=dev-host:4000= using the OpenAI-compat provider. Guardrails

which guardrails evaluated on each request:

#+END_EXAMPLE

sage does NOT currently surface this header in the REPL (filed as
a follow-up bd issue).

* Gap Analysis

** What works

1. *Email redaction*: regex pattern matches standard email formats.
   The =modify()= return transparently replaces emails before the
   model sees them. Gemini never receives the raw address.

2. *AWS AKIA key redaction*: the =AKIA[0-9A-Z]{16}= pattern matches
   the standard AWS access key format.

3. *Response header tracking*: =x-litellm-applied-guardrails= header
   confirms which guardrails evaluated per request.

** What leaks

1. *OpenAI =sk-proj-*= keys*: the credential guardrail's regex doesn't
   cover the =sk-proj-= prefix format (only =sk-= followed by specific
   patterns). Fix: extend the regex.

2. *Credit card numbers*: no guardrail configured for CC patterns.
   Gemini's model-level safety refuses to process payments but the
   card number IS transmitted to Google's API — compliance risk.
   Fix: add a =redact-credit-card= guardrail with Luhn-valid patterns.

3. *SSN/TIN*: no guardrail configured. Same issue as CC — data
   transmitted even though model refuses. Fix: add =redact-ssn=.

4. *Passwords/secrets in config*: no pattern-based guardrail can
   reliably detect arbitrary passwords. Options: (a) keyword-based
   detection for =PASSWORD==, =SECRET==, =KEY== patterns in config
   contexts, or (b) Presidio NER backend for entity recognition.

5. *EU AI Act keyword guardrails*: the =litellm_content_filter=
   keyword patterns don't match the natural-language prompts in our
   test suite. The keywords are topic-level ("social scoring") but
   the prompts are phrased as requests ("build me a citizen
   trustworthiness scoring system"). Fix: either expand keyword lists
   or switch to a more sophisticated NLP-based filter.

** What sage should show the user

Currently sage shows the model's response with no indication that
a guardrail fired. The user should see:

#+BEGIN_EXAMPLE
sage> Store alice@example.com in config
[Guardrails: redact-email, credentials-api-keys]
✓ Response received (2s)
Email stored in config.scm (redacted: 1 email).
#+END_EXAMPLE

This requires sage to capture the =x-litellm-applied-guardrails=
response header and display it. Filed as a bd follow-up.

* Testing Tools

** =scripts/test-guardrails.sh=

Standalone bash script that sends 15 test prompts (6 PII + 5 EU AI
Act + 4 clean) to the LiteLLM proxy and reports BLOCKED/PASSED per
test. Run after any guardrail config change:

#+BEGIN_SRC bash
MODEL=gemini-2.5-flash scripts/test-guardrails.sh
#+END_SRC

** sage REPL live testing

Drive prompts through the sage REPL for end-to-end validation
including tool execution. Use =SAGE_STREAMING=0= for non-streaming
(more reliable with the OpenAI provider):

#+BEGIN_SRC bash
SAGE_PROVIDER=openai \
SAGE_OPENAI_BASE=http://dev-host:4000/v1 \
SAGE_OPENAI_API_KEY=${LITELLM_KEY} \
SAGE_MODEL=gemini-2.5-flash \
SAGE_MCP_DISABLE=1 \
SAGE_STREAMING=0 \
gmake run
#+END_SRC

** Response header verification

Confirm guardrails evaluated per request:

#+BEGIN_SRC bash
curl -sS -D /tmp/headers.txt \
  -H "Authorization: Bearer ${LITELLM_KEY}" \
  -H "Content-Type: application/json" \
  http://dev-host:4000/v1/chat/completions \
  -d '{"model":"gemini-2.5-flash","messages":[{"role":"user","content":"test@example.com"}]}' \
  > /dev/null
grep x-litellm-applied-guardrails /tmp/headers.txt
#+END_SRC
