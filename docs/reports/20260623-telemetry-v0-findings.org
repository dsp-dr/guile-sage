* 1. The isolated test

Deliberately minimal — no LiteLLM, no real model call (avoid the
feature dot-product):

* 3. What's broken — and it is NOT guile-sage (CORRECTED 2026-06-23)

=lapdog= logs =received /v1/metrics payload with 0 resource metric(s)=
for guile-sage's (non-empty, well-formed) payload. *First diagnosis
(Content-Type missing) was WRONG* and is retracted — see below.

** 3.1 What guile-sage actually sends (verified, all correct)
   - =http-post-curl= (=src/sage/util.scm:464=) *already hardcodes*
     =-H 'Content-Type: application/json'=. The flush is not missing it.
   - Dumped payload: =resourceMetrics:1 → scopeMetrics:1 → metrics:2=,
     1489 bytes, =asInt= datapoints. Well-formed OTLP/JSON.
   - Field names are *lowerCamelCase* (=resourceMetrics=,
     =scopeMetrics=, =timeUnixNano=, =asInt=) — *exactly per the
     OTLP/JSON (proto3 JSON) encoding spec*.

** 3.2 The real root cause — a lapdog counter bug (snake vs camel)
   =ddapm_test_agent/agent.py:914=:
   =num_resource_metrics = len(metrics_data.get("resource_metrics", []))=
   It reads *=resource_metrics=* (snake_case — the *protobuf* field
   name) but OTLP/JSON on the wire is *=resourceMetrics=* (camelCase).
   - protobuf path: =protobuf_to_dict= yields snake_case → count works.
   - JSON path: =json.loads= keeps the wire's camelCase → =.get(
     "resource_metrics")= → =None= → *0*.
   So ANY spec-conformant OTLP/JSON metrics emitter shows "0" in
   lapdog's log. =decode_metrics_request= DOES accept the JSON (200,
   stored); only the count/log (and likely surfacing) uses the wrong
   key. Compounds with upstream #378 (OTLP accepted, not surfaced).

** 3.3 The fix: NONE in guile-sage
   guile-sage is OTLP/JSON-conformant. Do *not* change it to emit
   snake_case (that would be non-spec). The gap is lapdog's; raise it
   upstream *only via the user* (hard rule: no public issues from the
   agent). For our own dashboard, the bb/cljs receiver must read
   =resourceMetrics= (camelCase) — then guile-sage's metrics ingest
   correctly with zero emitter change.

* 4. lapdog as a contract-analysis instrument

lapdog records everything received — usable as a *wire-contract ledger*
for comparing what each harness emits:

| Source | What it gives |
|--------+---------------|
| =~/.lapdog/lapdog.log= | per-request: =received /v1/metrics payload with N resource metric(s)= + an =aiohttp.access= line (size, status, UA) for EVERY POST |
| =GET :8126/test/traces= (200) | captured APM traces (JSON) |
| =GET :8126/test/session/start= (200) | session snapshot/VCR boundary — the test-agent's record/replay mechanism |
| OTLP receivers =:4317/:4318= | accept + count resource metrics (the strict-parse signal that caught our gap) |

So: point each harness at lapdog, do one action, read the log — the
delta IS the harness's emitted contract.

* 5. The defined contracts to study next ("go back")

v0 surfaced that each layer has a *defined contract* we haven't checked
against:

1. *OTLP/HTTP* — https://opentelemetry.io/docs/specs/otlp/ : Content-Type
   (=application/json= vs =application/x-protobuf=), the
   =resourceMetrics → scopeMetrics → metrics= shape, sum/gauge
   temporality. (This is the v0 gap.)
2. *OTEL env-var spec* — =OTEL_EXPORTER_OTLP_ENDPOINT/_PROTOCOL=, etc.
   (see =mini-setup/docs/telemetry-env-contract.org= B1–B4).
3. *Per-harness lapdog wrapper contracts* — lapdog ships a dedicated
   integration module per harness:
   =backfill_claude.py=, =backfill_codex.py=, =backfill_pi.py=. Each is
   "a landed PR's worth" of wrapper support, encoding what that harness
   exposes and how lapdog hooks it (Claude Code plugin + hooks; Codex
   JSONL + proxy; Pi TS extension). *These are the reference contracts
   for adding guile-sage support* — read them to see the expected
   shape, then either emit-to-match or write a guile-sage backfill.
4. *Datadog ddtrace / LLMObs* — the =DD_*= + =TRACEPARENT= surface
   lapdog injects (see the telemetry-env-contract gist).
