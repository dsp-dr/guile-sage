The roadmap is driven by *where the system boundary sits*: what the LiteLLM
proxy normalizes for us (with leaks to watch) vs. what the *harness* (guile-sage)
must own. New features that change that boundary update this map first.

*Legend* — how much LiteLLM does for us:
- =N= Normalized — proxy provides a unified surface; use it directly.
- =S= Some — mapped, but imperfect (offline vs provider truth, per-provider).
- =L= Leaky — partial / provider-uneven / silent no-op on unsupported; verify.
- =H= Harness-owned — LiteLLM does NOT do this; guile-sage must build it.

Everything =N/S/L= is *below the LiteLLM line* (proxy provides; watch the leaks).
Everything =H= is *above the line* (the harness owns it). Boundary-changing
features move items across that line — re-check [[file:CHAOS-TESTING.org][CHAOS-TESTING.org]] /
[[file:SANDBOX-HARNESS-CONTRACT.org][SANDBOX-HARNESS-CONTRACT.org]] when they do.

* Strategic direction — be the MCP server, not just a client

The 2026-06 coding-agent review ([[file:reports/20260625-coding-agent-feature-review.org][report]]) makes the point sharp:
Claude Code and Codex are both MCP *clients*, and *Codex already runs as an MCP
server* (=codex mcp-server=). The winning move for guile-sage is not to match
their agent-UX feature-for-feature — it is to *expose sage's own surface (tool
registry, sandbox, eval, provenance/taint boundary) as an MCP server* so those
agents call sage. That flips guile-sage from "another coding agent" to "the
governed tool/sandbox substrate other agents run on" — and it reuses exactly the
L3 assets we already own. Building blocks are in place (=*tools*= +
=register-tool= with JSON schemas in =tools.scm=; the JSON-RPC + transport code
in =mcp.scm=); the gap is the server-side =tools/list=/=tools/call= loop.

** Dogfood scope (v1): two consumers, one stdio entrypoint

As the dogfood team we scope the MCP-server to the two hosts we actually use —
*Claude Code* and *Emacs* — and they want the *same* thing: a *stdio* MCP server.
So the whole feature is one entrypoint.

- *Build:* =(sage mcp-server)= — a stdio JSON-RPC loop handling =initialize=,
  =tools/list= (emit =*tools*= with their JSON schemas), =tools/call= (dispatch
  to the tool's exec-fn, honoring the =safe?= flag + the taint boundary). Pure
  stdin/stdout read-print loop — no subprocess fork, so it sidesteps the
  spawn-segfault boundary (see [[file:CHAOS-TESTING.org][CHAOS-TESTING.org]]).
- *Launch (stable command):* =gmake mcp-server= → e.g.
  =guile3 -L src -c '(use-modules (sage mcp-server)) (mcp-serve)'=.
- *Wire — Claude Code:*
  =claude mcp add guile-sage -- gmake -C /path/to/guile-sage mcp-server=
  (stdio; Claude spawns it and speaks JSON-RPC over stdin/stdout).
- *Wire — Emacs:* point an MCP client (e.g. =mcp.el= / =gptel= MCP) at the same
  stdio command =gmake -C /path/to/guile-sage mcp-server=.
- *Out of scope for v1:* HTTP/SSE transport, auth, remote hosting — add only if a
  third consumer needs them. Two clients, one stdio command, done.

Tracked: =SAGE-t2l=.

** Bidirectional MCP integration — milestone plan (Claude Code ↔ sage)

Two directions, two contracts:
- *OUTBOUND* — sage as *client* (=src/sage/mcp.scm=, [[file:MCP-CONTRACT.org][MCP-CONTRACT.org]]): sage
  calls external MCP servers.
- *INBOUND* — sage as *server* (=src/sage/mcp-server.scm=,
  [[file:MCP-SERVER-CONTRACT.org][MCP-SERVER-CONTRACT.org]]): Claude Code / Emacs call sage.

| M  | Milestone | Dir | Exit criteria | Status | Bead |
|----+-----------+-----+---------------+--------+------|
| M0 | sage MCP client | out | sage connects an external SSE server, registers its tools | *DONE* | — |

| M2 | inbound real (safe-only) | in | =sage= server serves =*tools*= (24 safe); B2 blocks unsafe; Claude Code connects | *DONE* (=sage= connected live; tool exercise pending) | =SAGE-t2l= |
| M3 | harden inbound | in | tools/call args validated vs =inputSchema= → =-32602=; limits | open | =SAGE-090= |
| M4 | red-team inbound | in | N1–N5 negatives automated (=gmake=), proven by runtime evidence | open | =SAGE-4yy= |
| M5 | bidirectional loop | both | a Claude-Code-invoked sage tool itself fans out via sage's MCP *client* (round trip both ways in one workflow) | open | =SAGE-o4y= |

Current state: inbound (Claude Code → sage) is *live and proven*; outbound shipped.
Next: M3 (harden) / M4 (red-team) before leaning on it, then M5 (the real
bidirectional loop) and M6 (Emacs + deploy).

* Notes

- This supersedes the old v0.1-era roadmap (removed 2026-06-24). It is a
  *boundary map*, not a feature checklist: the value is in the =Status= column,
  which says who owns each capability.
- The =L= rows are the active risk surface — each is a place a feature can
  silently no-op or diverge per provider. Treat them like the taint/guardrail
  boundaries: verify, don't assume (cf. the guardrail "applied ≠ redacted"
  finding in [[file:VALIDATION-INDEX.org::#live-runs][VALIDATION-INDEX]]).
- When a new feature moves an item across the LiteLLM line (e.g. a guardrail
  going from =L= to =N=, or a new =H= runtime capability), update this table and
  the affected contract docs in the same change.
