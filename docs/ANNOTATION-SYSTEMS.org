* 1. Why this exists
  :PROPERTIES:
  :CUSTOM_ID: why
  :STATUS: closed
  :END:

guile-sage's =docs/= tree is large (37 org/md files at top level alone,
plus =adr/=, =rfc/=, =reports/=). Many files make /claim-bearing/
assertions about the running system — module counts, default values,
port numbers, behavioral guarantees ("fire-and-forget", "never blocks"),
exported symbols, supported providers. The code is iterated faster than
the docs; the docs do not own a verifier; nothing reconciles them.

The wal.sh /Annotation Systems for Evolving Documents/ piece (Walsh,
2026-06-14) frames the same problem in a research-notebook context and
proposes a concrete primitive: org property drawers on every heading,
keyed by a stable =:CUSTOM_ID:=, carrying =:STATUS:=, =:VERDICT:=,
=:VERIFIED_BY:= and =:FINDING:= fields. Verifications append to a
tamper-evident JSONL chain. Verdicts are reconciled between drawer and
chain so they cannot silently diverge.

We adopt that contract, with three guile-sage-specific adjustments:

1. The /verifier of last resort/ is not a human reviewer — it is the
   guile-sage REPL itself running against a code grep, a test run, or
   an actual fetch. We split =:VERDICT:= (claim review by an agent or
   human) from =:EXEC_VERDICT:= (an executable assertion run by the
   harness).
2. Sub-agent attribution is mandatory because we routinely fan out
   independent verification (see [[#process]]).
3. Org headings live next to plain markdown (=AGENTS.md=, =CLAUDE.md=,
   =adr/*.md=). Property drawers are an org-only feature; for
   markdown, we use HTML-comment property blocks (see [[#schema-md]]).

* 2. Claim taxonomy
  :PROPERTIES:
  :CUSTOM_ID: taxonomy
  :STATUS: closed
  :END:

Not every sentence is a claim. Annotations attach only to
/empirically-falsifiable/ assertions. The architecture audit (§6) sorts
them into:

| Kind         | Example                                                |
|--------------+--------------------------------------------------------|
| ~path~       | "MCP config from =~/.claude.json="                     |
| ~symbol~     | "Ollama exports =ollama-chat-streaming="               |
| ~value~      | "Fast tier ceiling = 2000 tokens"                      |
| ~count~      | "21 source files"                                      |
| ~port~       | "OTel Collector at =:4318="                            |
| ~behavior~   | "Telemetry is fire-and-forget"                         |
| ~contract~   | "All MCP invocations require YOLO mode"                |
| ~vocabulary~ | "=:VERDICT:= ∈ {correct, corrected, disputed, …}"    |

Prose without one of these tags is /editorial/ and unannotated.

* 3. Schema — org headings
  :PROPERTIES:
  :CUSTOM_ID: schema-org
  :STATUS: closed
  :END:

Every annotated heading carries a =:CUSTOM_ID:= (stable, kebab-case,
**never updated when heading text changes** — the ID is the handle,
the heading text is the label). Annotations live in a property drawer
on the same heading:

#+BEGIN_SRC org
* OTel Collector receives OTLP/HTTP JSON
  :PROPERTIES:
  :CUSTOM_ID: otel-receiver
  :STATUS: closed
  :VERIFIED_AT: 2026-06-22T04:00Z
  :VERIFIED_BY: claude-agent
  :VERDICT: correct
  :EXEC_VERDICT: verified
  :EXEC_FINDING: grep src/sage/telemetry.scm shows derives :4318/v1/metrics from OTEL_EXPORTER_OTLP_ENDPOINT
  :END:
#+END_SRC

Field reference (from the wal.sh spec; we adopt verbatim):

| Field            | Type     | Meaning                                       |
|------------------+----------+-----------------------------------------------|
| =:CUSTOM_ID:=    | required | stable kebab-case slug, never renamed         |
| =:STATUS:=       | enum     | =open \| addressed \| closed=                 |
| =:REVIEW:=       | string   | "YYYY-MM-DD author: prose note" (optional)    |
| =:VERIFIED_AT:=  | ISO-8601 | last verification timestamp                   |
| =:VERIFIED_BY:=  | string   | agent or human identifier                     |
| =:VERDICT:=      | enum     | =correct \| corrected \| disputed \| needs-citation= |
| =:FINDING:=      | string   | one-line evidence (multi-line via =:FINDING+:=) |
| =:EXEC_VERDICT:= | enum     | =verified \| broken \| partial \| skipped=    |
| =:EXEC_FINDING:= | string   | command + result that executed the check      |

* 4. Schema — markdown
  :PROPERTIES:
  :CUSTOM_ID: schema-md
  :STATUS: open
  :REVIEW: 2026-06-22 claude-agent: shape proposed; tooling not yet written
  :END:

For =.md= files (=AGENTS.md=, =CLAUDE.md=, =adr/*.md=) we use an
HTML-comment block immediately after the heading:

#+BEGIN_SRC markdown
,## Decision: Ollama as first provider

<!-- :annotation:
custom-id: adr-0002-ollama
status: closed
verified-at: 2026-06-22T04:00Z
verified-by: claude-agent
verdict: correct
finding: src/sage/provider.scm defaults to 'ollama
-->
#+END_SRC

The HTML-comment is invisible to renderers and trivially greppable
(=grep -A 10 ":annotation:" docs/=).

* 5. Verification ledger
  :PROPERTIES:
  :CUSTOM_ID: ledger
  :STATUS: open
  :REVIEW: 2026-06-22 claude-agent: spec only — ledger writer not implemented
  :END:

Every verification append-writes one JSONL record to
=.verify/chain.jsonl=:

#+BEGIN_SRC json
{"target":"docs/ARCHITECTURE.org#module-summary","at":"2026-06-22T04:00Z","by":"claude-agent","verdict":"corrected","finding":"21 vs 27 actual","prev_sha":"abc…","sha":"def…"}
#+END_SRC

The =prev_sha= chains entries; =sha= is the SHA-256 of the canonicalised
record. =bd reconcile-ledger= (planned) compares drawer verdicts against
the ledger and fails CI if they diverge. Tamper-evidence is cheap
insurance against an agent silently rewriting a verdict in the drawer
without recording the change.

* 7. Process — sub-agent verification
  :PROPERTIES:
  :CUSTOM_ID: process
  :STATUS: closed
  :END:

The pattern that produced §6 is the one we are committing to:

1. /Author/ writes the doc (or the present-state of it is taken as
   input).
2. /Fan-out/: launch one or more independent sub-agents, each with a
   non-overlapping scope (one per major doc, or one per /claim kind/
   for the multi-axis case). Each sub-agent /must not/ see the
   author's prior conclusions.
3. /Each sub-agent/ emits a structured verdict per heading
   (=:VERDICT:=, =:FINDING:=, ideally =:EXEC_VERDICT:= via a real
   command). It MUST cite the verification command or grep target.
4. /Reconcile/: drawer entries written; one ledger row per verdict;
   conflicts surfaced as =:STATUS: open=.
5. /Close/: a follow-up agent (or the author) reads the open
   annotations and either fixes the doc, files a bead, or marks the
   claim =disputed= with rationale.

Attribution is part of the contract: =:VERIFIED_BY: claude-agent=,
=:VERIFIED_BY: vera-agent=, =:VERIFIED_BY: <human-handle>=. Multiple
verifications per heading are append, not overwrite — the property
drawer accumulates =:VERIFIED_AT:= rows in time order.

* 8. guile-sage-specific issues
  :PROPERTIES:
  :CUSTOM_ID: local-issues
  :STATUS: open
  :END:

What's harder /here/ than in the source piece:

** 8.1 Heterogeneous formats
   Org, markdown, mermaid, plantuml SRC blocks, JSON contract schemas
   (=docs/EVAL-CASE-SCHEMA.org=), C4 diagrams embedded in org. A single
   property-drawer convention covers org only; markdown needs the
   HTML-comment shim (§4); embedded diagrams cannot carry an attached
   annotation — they need their own heading wrapper.

** 8.2 Generated artefacts
   Some docs export to HTML/PDF (e.g. =RELEASE-*.org=).
   Annotations on a release-note heading are only
   meaningful pre-export; post-export, the snapshot is frozen.
   Annotations on release notes are =:STATUS: closed= by construction.

** 8.3 The verifier is the running system
   Unlike a citation graph, our verifications are operational: "the
   model tier ceiling is N" is verified by reading =src/sage/model-tier.scm=
   at HEAD. Three consequences:
   - A verdict has a /target git SHA/ that drifts as soon as someone
     edits the source. We have not yet encoded =:VERIFIED_AT_SHA:= in
     the schema but should.
   - Tests are also verifiers. A claim "all tests pass" is satisfied
     by the harness, not by re-grepping.
   - Some claims are unverifiable from inside the process — port
     reachability needs the network, contract conformance against the
     wal.sh spec needs the spec to be up. Those are =partial= /
     =skipped= EXEC verdicts.

** 8.4 Open gaps inherited from the source piece
   - /Multi-axis verdict roll-up/ is unspecified (per-axis verdicts
     yes; how to aggregate to a single =:VERDICT:= no).
   - /Verdict vocabulary drift/ is already present in our own
     adoption: we add =verified | broken | partial | skipped= for
     =:EXEC_VERDICT:= which is not in the source schema's drawer set.
     We accept the divergence and document it here.
   - /Validators go unvalidated/: nothing checks that the verification
     command in =:EXEC_FINDING:= /actually runs/ what the verdict
     claims. The ledger gives tamper-evidence but not validation.

* 10. Downstream methodology gaps (field report)
  :PROPERTIES:
  :CUSTOM_ID: methodology-gaps

These are /our notes, issues, and problems/ with the annotation methodology
itself — surfaced while doing the boundary/chaos/docs work, not drift in a
single doc. Each was demonstrated live (=:EXEC_VERDICT: verified=) and is
tracked upstream in =jwalsh/www.wal.sh= (=walsh-kx1=). NOTE: that repo does not
yet exist on GitHub; =ghq get jwalsh/www.wal.sh= reports "exists" only because a
local placeholder tracker was created — it has no remote until the repo is made.

** 10.1 History rewrites orphan SHA-keyed annotations
   :PROPERTIES:
   :CUSTOM_ID: gap-history-rewrite
   :STATUS: open
   :VERDICT: disputed
   :FINDING: §6.4 (version-aware annotations via git), §3/§8.2 (git notes) and the JSONL chain assume append-only history. A filter-repo --replace-text force-push changes every commit hash, orphaning all SHA-keyed annotations and detaching notes. Demonstrated: origin/main rewritten to scrub mac.lan->host.lan, zero common ancestor with prior clones (RCA gist fef45e01c0da095ea4d5dc7b0a0f4337). Anchor the chain to content/CUSTOM_ID hashes, not commit SHAs.
   :EXEC_VERDICT: verified
   :FILED: walsh-kx1
   :END:

** 10.2 EXEC_VERDICT conflicts with the de-hardcoded-IP lint
   :PROPERTIES:
   :CUSTOM_ID: gap-exec-vs-lint
   :STATUS: open
   :VERDICT: disputed
   :FINDING: §8.3 makes the running system the verifier, but :EXEC_VERDICT: assertions that must hit a concrete host cannot embed it — gmake lint-docs forbids hardcoded RFC1918 IPs in living docs (the very rule whose retroactive enforcement caused 10.1). §11.6 (point-in-time/single-host) is adjacent, not this. Resolution: EXEC_VERDICT references a parameterized $HOST; resolved value + result recorded in the chain, never the doc.
   :EXEC_VERDICT: verified
   :FILED: walsh-kx1
   :END:

** 10.3 Cross-repo / external-service claims have no anchor home
   :PROPERTIES:
   :CUSTOM_ID: gap-foreign-claims
   :STATUS: open
   :VERDICT: disputed
   :FINDING: Annotations anchor within one doc tree. This session's verification spanned guile-sage docs + the wal.sh page + a live LiteLLM gateway (claim "team key returns SAGE_OK"). External-service / other-repo claims have no :CUSTOM_ID: home and no stable verifier. Proposal: a "foreign claim" form keyed by URL + content-hash of the response, in a separate annotations file.
   :EXEC_VERDICT: verified
   :FILED: walsh-kx1
   :END:

** 10.4 git-notes fragility under rebase + non-default push
   :PROPERTIES:
   :CUSTOM_ID: gap-git-notes
   :STATUS: open
   :VERDICT: disputed
   :FINDING: §3/§8.2 present git notes favorably. In practice our first-principles notes detached when rebase changed commit hashes, and notes do not travel without an explicit refs/notes/* push. Add a caveat; relates to 10.1.
   :EXEC_VERDICT: verified
   :FILED: walsh-kx1
   :END:

** 10.5 Tooling friction observed (not methodology, but logged)
   :PROPERTIES:
