** Overview

guile-sage is an interactive AI agent REPL written in GNU Guile 3 Scheme. It provides:

- Multi-provider LLM support (Ollama, Gemini, LiteLLM/OpenAI-compat)
- 33+ built-in tools with safe/unsafe permission model
- Multi-step tool chain execution with signature-aware loop detection
- =fetch_url= with content negotiation (markdown preferred), XML envelope
  (source + SHA-256 + content-type + trust), and automatic scratch-store
  offloading for large bodies
- =scratch_get= / =scratch_list= for paged retrieval of large tool outputs
  without blowing the context window
- LIFO =sage_task_push= alongside FIFO =sage_task_create= for agents
  decomposing work into ordered sub-steps
- MCP client (SSE transport) for external tool servers, with =/mcp=
  endpoint table and optional health/metrics probing
- Lifecycle hooks (=PreToolUse= / =PostToolUse=) with veto semantics,
  error isolation, and a pre-wired observer mode (=SAGE_DEMO_HOOKS=1=)
- =eval_scheme= sandbox (denylist on shell-out, pipes, filesystem
  mutation, code loading) that fires even in YOLO mode
- Session management with persistence and context compaction (5 strategies)
- Local JSONL usage ledger (=/stats=) at =$XDG_STATE_HOME/sage/usage.jsonl=
- OTLP telemetry emission to Prometheus/Grafana (opt-in)
- Readline-coloured prompt with token-budget indicator
- Unified-diff =edit_file= summaries (model and user see the same change)
- Image generation via Ollama flux models
- LiteLLM guardrail integration (EU AI Act compliance)

*** Tool System

Tools are classified as /safe/ (always allowed) or /unsafe/ (require YOLO mode).
The default configuration enables most development tools as safe for interactive use.

| Tool | Description | Safe |
|------+-------------+------|
| =read_file= | Read file contents | Yes |
| =list_files= | List directory contents | Yes |
| =write_file= | Write content to file | No (YOLO) |
| =edit_file= | Search/replace returning a unified diff | No (YOLO) |
| =glob_files= | Find files by pattern | Yes |
| =search_files= | Grep for patterns | Yes |

**** Network + Content
| Tool | Description | Safe |
|------+-------------+------|
| =fetch_url= | GET an http(s) URL; returns an XML envelope with source, content-type, SHA-256, trust label, and body (inline under 1 KB, scratch-stored otherwise) | Yes |
| =scratch_get= | Paged retrieval of scratch-stored content by SHA | Yes |
| =scratch_list= | Enumerate scratch-stored SHAs with byte counts | Yes |

| =git_status= | Show git status | Yes |

| =git_add_note= | Add git notes | No (YOLO) |

| Tool | Description | Safe |
|------+-------------+------|

**** Agent Tasks
| Tool | Description | Safe |
|------+-------------+------|
| =sage_task_create= | Append task (FIFO) | Yes |
| =sage_task_push= | Push sub-task (LIFO) — runs BEFORE queued siblings | Yes |
| =sage_task_list= | List pending tasks | Yes |
| =sage_task_complete= | Mark current task complete | Yes |
| =sage_task_status= | Get agent mode + iteration count | Yes |

**** Meta/Unsafe (require YOLO mode)
| Tool | Description | Safe |
|------+-------------+------|
| =run_tests= | Execute test suite | No |
| =eval_scheme= | Evaluate Scheme code (denylist-sandboxed even in YOLO — blocks =system=, =open-*-pipe=, filesystem mutation, =load=, nested =eval-string=) | No |
| =reload_module= | Hot-reload modules | No |
| =create_tool= | Register new tools at runtime | No |
| =generate_image= | Generate an image via Ollama flux | No |

Enable YOLO mode for unsafe tools: =SAGE_YOLO_MODE=1 make run=. The
=eval_scheme= sandbox stays on in YOLO mode by design (NC12 in
=docs/HOOK-NEGATIVE-CONTRACTS.org=).

** Responsible Use

guile-sage is a *permissive* AI coding agent by design. Several features
shift safety decisions onto the operator:

- =SAGE_YOLO_MODE=1= disables the safe/unsafe tool separation documented
  in =docs/adr/0003-security-model.md=. In YOLO mode the agent can
  write files, commit, push to remotes, and (subject to the sandbox
  below) evaluate Scheme.
- The =eval_scheme= tool runs in a *denylist sandbox* that rejects
  =system=, =open-input-pipe=, =open-output-pipe=, =delete-file=,
  =rename-file=, =chmod=, =primitive-load=, =load=, nested =eval-string=,
  and other shell-out or filesystem-mutation primitives. The sandbox
  fires even in YOLO mode (NC12 in =docs/HOOK-NEGATIVE-CONTRACTS.org=).
  Safe arithmetic / string / list operations still work. This is a
  denylist, not a hermetic jail — treat it as defence-in-depth, not
  isolation.
- Autonomous modes (=agent-mode= interactive/autonomous/yolo) allow
  long-running multi-step actions without human confirmation per step.
- Tool results are framed with a =<tool-result>= role-boundary wrapper
  and a system-prompt clause telling the model to treat the contents
  as DATA, not instructions. This mitigates (does not eliminate)
  prompt-injection via file contents or fetched pages.

*You, the operator, are responsible for:*
- Running sage on systems whose blast radius you accept
- Vetting prompts before autonomous runs
- Not connecting sage to production credentials or systems you cannot
  afford to lose
- Reviewing generated code before merging or deploying
- Treating model output as untrusted input (see =docs/adr/0003-security-model.md=)

*Recommended starting posture:*
- Leave =SAGE_YOLO_MODE= unset (safe tools only)
- Use =interactive= agent mode until you trust a given task
- Point sage at a scratch workspace, not your primary checkout
- Review all diffs before commit

** Requirements

- *GNU Guile 3.0+* (=guile= on macOS/Linux, =guile3= on FreeBSD)
- *curl* — used for =fetch_url= and for streaming HTTP to the LLM
  provider (Gemini, Ollama). A future refactor (bead =guile-sage-szw=)
  may migrate the non-streaming path to =(web client)= + =guile-gnutls=
  but curl stays required today.
- *An LLM backend*, one of:
  - *Ollama* running locally (=localhost:11434= by default) — zero-cost,
    no API keys; suitable for vetting with =qwen3= / =qwen2.5-coder=
    / =deepseek-coder-v2= / =llama3.2=.
  - *Gemini API* (=GEMINI_API_KEY=) for cloud inference.
  - *LiteLLM / OpenAI-compat proxy* (=SAGE_OPENAI_BASE= +
    =SAGE_OPENAI_API_KEY=) for mixed routing with guardrails.

Optional:

- *guile-gnutls* — needed only if you use the experimental native HTTPS
  path via =(web client)=. Not required for the default curl-based
  flow.
- *git-filter-repo* — only if you're maintaining a fork and need to
  rewrite history.
- *asciinema* + *agg* — only for =make demo= (README GIF rebuild).

** Configuration

Create a =.env= file based on the template:

#+begin_src bash
cp .env.template .env
#+end_src

Configure your Ollama endpoint:

#+begin_src bash
# Local Ollama
SAGE_OLLAMA_HOST=http://localhost:11434
SAGE_MODEL=qwen3-coder:latest

# Cloud Ollama (with authentication)
SAGE_OLLAMA_HOST=https://api.example.com
SAGE_OLLAMA_API_KEY=your-api-key
#+end_src

*** XDG Directory Structure

guile-sage follows the XDG Base Directory specification:

#+begin_example
$XDG_CONFIG_HOME/sage/     # ~/.config/sage
├── .env                   # Optional: per-user dotenv (loaded at startup)
└── commands.scm           # Custom slash commands

$XDG_DATA_HOME/sage/       # ~/.local/share/sage
├── sessions/              # Saved conversation sessions
├── agents/                # Custom agent definitions
├── projects/              # Project-specific configurations
└── commands/              # Custom slash commands

$XDG_CACHE_HOME/sage/      # ~/.cache/sage
└── (cache files)

$XDG_STATE_HOME/sage/      # ~/.local/state/sage
└── (state files)
#+end_example

*** Optional: OTLP Telemetry (new in 0.6.0)

guile-sage can emit OpenTelemetry counters (sessions, tokens, tool
decisions, active time) to any OTel Collector. Enable by setting
=OTEL_EXPORTER_OTLP_METRICS_ENDPOINT= in any of the three places
sage looks for config — shell init, =~/.config/sage/.env=, or the
project-local =.env=. See =docs/TELEMETRY.org= for the full setup
and verification harness.

Override with environment variables:

#+begin_src bash
SAGE_CONFIG_DIR=/custom/config/path
SAGE_DATA_DIR=/custom/data/path
SAGE_SESSION_DIR=/custom/sessions/path
#+end_src

*** Project-Local Sessions

Like Claude Code, guile-sage stores sessions per-project using path encoding:

#+begin_example
/home/user/myproject  ->  -home-user-myproject
#+end_example

Sessions are stored in:
#+begin_example
$XDG_DATA_HOME/sage/projects/-home-user-myproject/sessions/
#+end_example

This keeps conversation history separate per working directory.

** Architecture

*** Beads Methodology

This project follows the beads methodology for development:
- Small, atomic commits
- Progressive commit protocol (PCP)
- Git notes for documentation
- Explicit file staging (never =git add -A=)

*** Self-Modification

guile-sage can modify itself through its tool system:
- =eval_scheme= evaluates arbitrary Scheme code
- =create_tool= registers new tools dynamically
- =reload_module= hot-reloads modified modules
- =edit_file= modifies source files

** Platform Notes

*** macOS (primary dev target)

- Homebrew Guile 3.0.11. HTTPS via =(web client)= requires
  =brew install guile-gnutls= (sets =GUILE_TLS_CERTIFICATE_DIRECTORY=).
- =system*= and =open-input-pipe= fail with "Bad file descriptor" under
  Guile 3.0.11 on macOS (Darwin spawn bug). sage uses a custom
  =capture-argv-in-dir= helper (primitive-fork + execlp) for all
  subprocess spawns; no =/bin/sh= is involved.
- Tested on Darwin 25.x (Tahoe). arm64 only; x86_64 untested.

*** FreeBSD

Known to run on FreeBSD 14.3. The argv-based subprocess path is also
used here (no platform-specific workaround needed). =guile-json= is
intentionally not used — the project ships a small parser in
=src/sage/util.scm=.

*** Linux

Expected to work on any distro with Guile 3.0 + curl. Not routinely
tested; report issues.
