** Overview

guile-sage is an interactive AI agent REPL written in GNU Guile 3 Scheme. It provides:

- Multi-provider LLM support (Ollama, Gemini, LiteLLM/OpenAI-compat)
- 22+ built-in tools with safe/unsafe permission model
- Multi-step tool chain execution with degenerate loop detection
- MCP client (SSE transport) for external tool servers
- Session management with persistence and context compaction (5 strategies)
- OTLP telemetry emission to Prometheus/Grafana
- Image generation via Ollama flux models
- LiteLLM guardrail integration (EU AI Act compliance)

*** Tool System

Tools are classified as /safe/ (always allowed) or /unsafe/ (require YOLO mode).
The default configuration enables most development tools as safe for interactive use.

| Tool | Description | Safe |

| Tool | Description | Safe |

** Responsible Use

guile-sage is a *permissive* AI coding agent by design. Several features
shift safety decisions onto the operator:

- =SAGE_YOLO_MODE=1= disables the safe/unsafe tool separation documented
  in =docs/adr/0003-security-model.md=. In YOLO mode the agent can
  write files, commit, push to remotes, and evaluate arbitrary Scheme.
- The =eval_scheme= tool evaluates arbitrary expressions with the REPL's
  process privileges. There is no sandbox — it can read =.env=,
  shell out, or exfiltrate data if an attacker controls the prompt.
- Autonomous modes (=agent-mode= interactive/autonomous/yolo) allow
  long-running multi-step actions without human confirmation per step.

*You, the operator, are responsible for:*
- Running sage on systems whose blast radius you accept
- Vetting prompts before autonomous runs
- Not connecting sage to production credentials or systems you cannot
  afford to lose
- Reviewing generated code before merging or deploying
- Treating model output as untrusted input (see =docs/adr/0003-security-model.md=)

*Recommended starting posture:*
- Leave =SAGE_YOLO_MODE= unset (safe tools only)
- Use =interactive= agent mode until you trust a given task
- Point sage at a scratch workspace, not your primary checkout
- Review all diffs before commit

** Configuration

Create a =.env= file based on the template:

#+begin_src bash
cp .env.template .env
#+end_src

Configure your Ollama endpoint:

#+begin_src bash
# Local Ollama
SAGE_OLLAMA_HOST=http://localhost:11434
SAGE_MODEL=qwen3-coder:latest

# Cloud Ollama (with authentication)
SAGE_OLLAMA_HOST=https://api.example.com
SAGE_OLLAMA_API_KEY=your-api-key
#+end_src

*** XDG Directory Structure

guile-sage follows the XDG Base Directory specification:

#+begin_example
$XDG_CONFIG_HOME/sage/     # ~/.config/sage
├── .env                   # Optional: per-user dotenv (loaded at startup)
└── commands.scm           # Custom slash commands

$XDG_DATA_HOME/sage/       # ~/.local/share/sage
├── sessions/              # Saved conversation sessions
├── agents/                # Custom agent definitions
├── projects/              # Project-specific configurations
└── commands/              # Custom slash commands

$XDG_CACHE_HOME/sage/      # ~/.cache/sage
└── (cache files)

$XDG_STATE_HOME/sage/      # ~/.local/state/sage
└── (state files)
#+end_example

*** Optional: OTLP Telemetry (new in 0.6.0)

guile-sage can emit OpenTelemetry counters (sessions, tokens, tool
decisions, active time) to any OTel Collector. Enable by setting
=OTEL_EXPORTER_OTLP_METRICS_ENDPOINT= in any of the three places
sage looks for config — shell init, =~/.config/sage/.env=, or the
project-local =.env=. See =docs/TELEMETRY.org= for the full setup
and verification harness.

Override with environment variables:

#+begin_src bash
SAGE_CONFIG_DIR=/custom/config/path
SAGE_DATA_DIR=/custom/data/path
SAGE_SESSION_DIR=/custom/sessions/path
#+end_src

*** Project-Local Sessions

Like Claude Code, guile-sage stores sessions per-project using path encoding:

#+begin_example
/home/user/myproject  ->  -home-user-myproject
#+end_example

Sessions are stored in:
#+begin_example
$XDG_DATA_HOME/sage/projects/-home-user-myproject/sessions/
#+end_example

This keeps conversation history separate per working directory.

** Architecture

*** Beads Methodology

This project follows the beads methodology for development:
- Small, atomic commits
- Progressive commit protocol (PCP)
- Git notes for documentation
- Explicit file staging (never =git add -A=)

*** Self-Modification

guile-sage can modify itself through its tool system:
- =eval_scheme= evaluates arbitrary Scheme code
- =create_tool= registers new tools dynamically
- =reload_module= hot-reloads modified modules
- =edit_file= modifies source files

** Platform Notes

*** FreeBSD

Developed and tested on FreeBSD 14.3. Known workarounds:
- Uses =curl= with temp files instead of =open-input-pipe= (segfault issue)
- Custom JSON parser (guile-json not available)
